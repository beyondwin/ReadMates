## 휴먼 측정 패스 절차

### 1. 베이스라인 (코드 변경 전 상태)

> 주의: 현재 브랜치는 변경이 이미 적용된 상태입니다. 베이스라인은 **변경 전 commit**에서 채취해야 합니다.

```bash
# 변경 전 SHA로 임시 브랜치 만들고 측정
git checkout -b bench/baseline 55407e6   # 본 변경 직전 SHA
scripts/bench/measure-local.sh baseline cold
scripts/bench/measure-local.sh baseline warm

# CI 베이스라인 (의도적인 노이즈 commit 3회 push)
git commit --allow-empty -m "bench: baseline run 1" && git push
# (run 2, 3 반복 후 gh run view로 backend/frontend/e2e duration 채취)

# 최적화 브랜치로 복귀
git checkout readmates-build-test-speed-20260516-123042
```

### 2. 적용 후 측정 (각 Task별로 누적)

각 Task의 commit에 도달한 시점에서 다음을 실행하면 Task별 효과 귀속이 가능합니다:

```bash
git checkout 320e4e4 && scripts/bench/measure-local.sh after-task1 warm
git checkout d02eaee && scripts/bench/measure-local.sh after-task2 warm
# ... (after-task3, after-task4, after-task5)
```

cold 측정은 시간 부담이 크면 baseline + after-task5만 채취해도 spec §7.1을 채울 수 있습니다.

### 3. Fork 수 sweep (선택)

```bash

# docs/superpowers/reports/2026-05-16-sweep-forks.md 결과 확인
# 최단 median을 .github/workflows/ci.yml의 READMATES_TEST_FORKS 값에 반영
```

### 4. CI 측정

```bash
# 최적화 브랜치 push 후
gh run list --workflow=ci.yml --branch=readmates-build-test-speed-20260516-123042 \
  --json databaseId,conclusion,createdAt --limit 3
gh run view <run-id> --json jobs \
  --jq '.jobs[] | {name, conclusion, startedAt, completedAt}'
```

3회 run의 `Backend`/`Frontend`/`E2E (1/3)` duration을 산출해
`docs/superpowers/reports/2026-05-16-after-final-ci.md`에 저장.

### 5. Spec §7 표 채우기

`scripts/bench/measure-local.sh`가 만든 reports 파일의 median 값을 spec §7.1/7.2/7.3/7.4의 빈 셀에 옮겨 적고, Δ% 컬럼은 `(After - Before)/Before * 100`으로 계산.

§5 A3 (`C1 ≥ 30% 단축`), A4 (`L1 warm ≥ 25% 단축`)를 §7 표 아래 자체 검증으로 PASS/FAIL 마킹.

## 즉시 검증 가능한 항목 (측정 불필요)

### A1: `:test` 미실행 확인

```bash
cd server && ./gradlew check --dry-run 2>&1 | grep -c ':test\b'
# :test가 그래프에는 남지만 enabled=false. 직접 실행으로 무동작 확인:
./gradlew :test
# Expected: "BUILD SUCCESSFUL" with no JUnit XML output, 4 actionable tasks (compile chain only).
ls -la server/build/test-results/test/ 2>/dev/null || echo "No :test results dir = OK"
```

### A2: unit class 카운트 일치

```bash
cd server
EXPECTED_UNIT=$(grep -rL '@Tag("integration")\|@Tag("container")\|@Tag("architecture")' \
  src/test/kotlin --include='*Test.kt' | wc -l | tr -d ' ')
./gradlew clean unitTest
UNIT_RAN=$(ls build/test-results/unitTest/TEST-*.xml 2>/dev/null | wc -l | tr -d ' ')
echo "expected=$EXPECTED_UNIT actual=$UNIT_RAN"
```

기대값: 두 숫자 동일 (예: 58 vs 58).

### A5, A6: 기존 게이트 통과

```bash
cd server && ./gradlew check architectureTest integrationTest
```

모두 BUILD SUCCESSFUL이면 PASS.

### A7: E2E 영향 없음

CI 푸시 후 `gh pr checks` 또는 `gh run view`로 `E2E (1/3..3/3)` conclusion=success 확인.

## 알려진 제약

- **Task 3 (forks) 최적값 미측정**: 코드 default는 `availableProcessors()/2`. GHA `ubuntu-latest`(4 vCPU)에서는 자동으로 `2`가 나오며, spec §4.3의 "일반적으로 2 fork가 sweet spot" 휴리스틱과 일치. 머신별 측정으로 더 좋은 값이 나오면 `READMATES_TEST_FORKS` env로 override.
- **Task 4 audit 한계**: companion object 28건 중 unit-tagged 3건은 코드 검사로 mutable 상태 없음을 확인했으나, 자동 실행기는 3회 연속 PASS flaky 검증을 수행하지 않았습니다. 휴먼 측정 패스에서 `for i in 1 2 3; do ./gradlew clean unitTest || break; done`을 권장.
- **Task 5 (reuse) 검증 없음**: `withReuse(true)` 적용 후 실제로 `~/.testcontainers.properties` 옵트인 한 상태에서 컨테이너 재시작이 건너뛰어지는지는 휴먼 측정에서 확인. `withReuse(true)` 자체가 컴파일/실행에 영향 없는 마킹이므로 risk는 낮음.

## 헤드리스 자동 실행 시 마주친 제약 (학습 기록)

`kws-claude-multi-agent-executor`가 본 plan을 헤드리스로 실행하려 시도했으나 Phase 0에서 halt했습니다. 사유:

1. Plan이 `## Task N:` (level-2) markdown 헤더 사용 — 스킬은 `### Task N:` (level-3) 요구.
2. Plan 아키텍처가 "measure → change → measure → gate → commit" 사이클로 설계 — multi-hour wallclock + `git push` × 3 / task가 단일 헤드리스 세션의 컨텍스트 + 권한 범위 초과.

대응:
- 헤더 이슈는 표면적인 mechanical 문제이며, 자동 실행기를 가능하면 위해 향후 plan은 `### Task N:` 사용.
- 측정 주도 plan은 `superpowers:subagent-driven-development` 인터랙티브 실행 또는 split-plan 패턴이 적합. 현재 작업은 코드 변경만 인라인으로 실행하고 측정을 본 핸드오프 노트로 분리하는 path로 마무리.
