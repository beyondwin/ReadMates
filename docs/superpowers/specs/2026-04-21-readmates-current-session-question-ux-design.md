# ReadMates Current Session Question UX Design

작성일: 2026-04-21
상태: VALIDATED DESIGN SPEC
문서 목적: `/app/session/current`의 질문 등록 흐름을 단순화하고, 모임 이후 기록 섹션의 순서와 시각 톤을 정리하기 위한 UX, API, 상태, 테스트 기준을 정의한다.

## 1. 문제 정의

현재 세션 화면의 질문 등록은 `Q1~Q5` 우선순위 버튼을 먼저 고른 뒤, 선택된 하나의 질문과 `초안 생각 · 선택`을 작성하고 저장하는 방식이다.

이 흐름은 사용자가 "질문을 여러 개 남긴다"는 실제 작업보다 "번호를 선택하고, 화면이 바뀌는지 기억하고, 각 번호를 따로 저장한다"는 조작에 더 많은 주의를 쓰게 만든다. 특히 모바일에서는 같은 카드 안에 우선순위 칩, 질문 입력, 초안 입력, 상태 문구, 저장 버튼이 함께 있어 작성 흐름이 길고 복잡하게 느껴진다.

또한 사용자는 초안 생각 선택 입력을 삭제하길 원한다. 이 입력은 현재 질문 저장 payload와 아카이브 조회에는 남아 있지만, 현재 준비 화면의 핵심 작업에서는 제거해도 된다.

모임 이후 기록 영역에서는 한줄평이 서평보다 뒤에 있고, 한줄평 카드가 `surface-quiet`/`m-card-quiet` 톤으로 보여 서평보다 덜 중요한 보조 영역처럼 보인다. 사용자는 한줄평과 모임 이후 서평의 위치를 바꾸고, 한줄평의 회색 톤을 서평과 동일한 톤으로 맞추길 원한다.

## 2. 목표

- 질문 작성 과정을 데스크탑과 모바일 모두에서 한 화면의 리스트형 편집으로 단순화한다.
- 사용자가 우선순위를 직접 선택하지 않아도 위에서부터 질문 순서가 정해지게 한다.
- 질문은 최소 2개, 최대 5개까지 작성 가능하게 한다.
- `초안 생각 · 선택` 입력을 현재 세션 질문 작성 UI에서 제거한다.
- 질문 저장 시 서버 상태가 화면의 질문 목록과 일치하게 한다.
- 한줄평 섹션을 서평 섹션보다 먼저 배치한다.
- 한줄평 섹션의 배경과 버튼 톤을 서평 섹션과 동일하게 맞춘다.
- 기존 개별 질문 저장 API와 저장된 과거 데이터는 깨지지 않게 유지한다.
- 변경 범위를 현재 세션 준비 화면과 필요한 질문 저장 API에 한정한다.

## 3. 비목표

- 질문에 대한 별도 드래그 앤 드롭 정렬은 추가하지 않는다.
- 질문별 공개 범위, 댓글, 투표, 추천 기능은 추가하지 않는다.
- 아카이브 화면에서 기존 `draftThought` 표시 정책을 이번 변경에서 제거하지 않는다.
- DB 컬럼 `questions.draft_thought`를 삭제하지 않는다.
- 현재 세션 화면 전체 정보 구조를 다시 설계하지 않는다.
- 한줄평과 장문 서평 저장 API의 도메인 정책은 바꾸지 않는다.

## 4. 확정된 방향

질문 작성은 "우선순위 선택형 단일 편집기"에서 "리스트형 다중 질문 편집기"로 바꾼다.

초기 렌더링:

- 저장된 내 질문이 2개 이상이면 저장된 질문 수만큼 입력칸을 보여준다.
- 저장된 내 질문이 0개 또는 1개이면 입력칸을 최소 2개까지 채워 보여준다.
- 저장된 질문은 priority 오름차순으로 정렬해 입력칸에 배치한다.

작성 흐름:

- 입력칸 라벨은 `질문 1`, `질문 2`, `질문 3`, `질문 4`, `질문 5`를 사용한다.
- 사용자는 `Q1~Q5` 탭을 누르지 않는다.
- 위에서부터 입력칸 순서가 저장 priority가 된다.
- `질문 추가` 버튼으로 최대 5개까지 입력칸을 늘린다.
- 질문이 5개가 되면 추가 버튼은 비활성화하고 최대 5개 안내를 보여준다.
- `질문 1`과 `질문 2`는 최소 개수이므로 삭제할 수 없다.
- `질문 3~5`는 삭제 버튼으로 제거할 수 있다.
- 삭제 후 남은 질문은 위에서부터 다시 `질문 1~N`으로 번호가 정리된다.

저장 흐름:

- `질문 저장` 버튼 하나가 현재 보이는 질문 목록 전체를 저장한다.
- 비어 있거나 공백뿐인 입력칸은 유효한 질문으로 세지 않는다.
- 유효 질문이 2개 미만이면 네트워크 요청 없이 인라인 오류를 보여준다.
- 유효 질문이 2~5개이면 서버에 전체 목록을 교체 저장한다.
- 저장 성공 후 기존처럼 `router.refresh()`를 호출해 공동 보드와 상태를 갱신한다.

## 5. 데스크탑 UX

데스크탑의 `이번 달 내 질문` 카드는 다음처럼 바꾼다.

- eyebrow: `Question`
- 제목: `이번 달 내 질문`
- 보조 문구: `최소 2개, 최대 5개까지 준비해 주세요.`
- 오른쪽 상태: `내 질문 N/5`
- 본문: 질문 입력 행 리스트
- 하단: `질문 추가`, 저장 피드백, `질문 저장`

각 질문 입력 행은 다음 요소를 가진다.

- 왼쪽 라벨: `질문 1`
- 오른쪽 삭제 버튼: `질문 3~5`에서만 노출
- textarea: 질문 본문

textarea placeholder는 짧게 유지한다.

- `모임에서 나누고 싶은 질문을 적어 주세요.`

삭제 버튼은 텍스트 버튼보다 작은 아이콘 버튼 또는 `삭제` 보조 버튼으로 표현한다. 기존 디자인 시스템에 별도 아이콘 버튼 패턴이 부족하면 `btn btn-ghost btn-sm`을 사용하되, 버튼의 accessible name은 `질문 3 삭제`처럼 구체적으로 둔다.

데스크탑 질문 카드는 기존 `surface` 톤을 유지한다. 초안 생각 라벨과 textarea는 제거한다.

## 6. 모바일 UX

모바일 `내 준비` 탭의 질문 작성 카드는 데스크탑과 같은 리스트형 모델을 사용한다.

- 상단 eyebrow: `질문 작성`
- 오른쪽 상태: `N/5`
- 본문: 질문 입력 행 리스트
- 하단: `질문 추가`, 저장 피드백, `질문 저장`

모바일에서는 입력 행 간 간격을 충분히 두고, 각 textarea는 최소 3줄을 유지한다. 삭제 버튼은 오른쪽 상단에 작게 배치하되, 터치 목표가 너무 작아지지 않게 기존 버튼 높이를 따른다.

모바일에서도 `초안 생각 · 선택` 라벨과 textarea는 제거한다.

## 7. 모임 이후 기록 섹션

데스크탑 개인 준비 컬럼의 모임 이후 기록 순서는 다음으로 바꾼다.

1. 한줄평
2. 모임 이후 · 서평

모바일 `내 기록` 탭의 순서도 동일하게 바꾼다.

한줄평 카드 톤:

- 데스크탑 한줄평 섹션은 `surface-quiet` 대신 `surface`를 사용한다.
- 모바일 한줄평 섹션은 `m-card-quiet` 대신 `m-card`를 사용한다.
- 데스크탑 한줄평 저장 버튼은 `btn btn-quiet btn-sm` 대신 `btn btn-primary btn-sm`를 사용해 서평 저장 버튼과 같은 위계를 가진다.
- 한줄평 badge, input, 저장 피드백 위치는 기존 구조를 유지한다.

서평 카드의 문구와 톤은 유지하되, 한줄평 아래로 이동한다.

## 8. Frontend State Design

현재 `QuestionDraft`는 `text`와 `draftThought`를 담는다. 새 UI에서는 현재 세션 질문 작성 상태를 질문 본문 배열로 단순화한다.

권장 상태:

```ts
type QuestionInput = {
  clientId: string;
  text: string;
};
```

`clientId`는 React key 안정성을 위해 프런트에서만 사용한다. 서버로 보내지 않는다.

초기화:

- `session.myQuestions`를 priority 오름차순으로 정렬한다.
- 각 질문을 `{ clientId, text }`로 변환한다.
- 결과 길이가 2보다 작으면 빈 입력을 추가해 2개로 맞춘다.
- 결과 길이가 5보다 크면 서버 정책상 이상 상태이므로 앞의 5개만 렌더링하고 저장 시 5개만 보낸다.

파생 상태:

- `validQuestions`: `text.trim()`이 비어 있지 않은 입력
- `writtenQuestionCount`: `validQuestions.length`
- `canAddQuestion`: `questionInputs.length < 5`
- `canRemoveQuestion`: `questionInputs.length > 2`이고 대상 index가 2 이상
- `questionValidationMessage`: 유효 질문이 2개 미만일 때만 저장 시 표시

상태 카드:

- 데스크탑 `MyStatusCard`와 모바일 `MobilePrepMeta`에서 `초안` 항목을 제거한다.
- 질문 항목의 완료 기준은 `writtenQuestionCount >= 2`다.
- 값은 `${writtenQuestionCount}/5`를 유지하되, 완료 전에는 보조 문구나 색상으로 미완료 상태를 보여준다.

## 9. Backend API Design

기존 API:

- `POST /api/sessions/current/questions`
- `{ priority, text, draftThought }`

기존 API는 호환성을 위해 유지한다. 새 UX는 새 컬렉션 교체 API를 사용한다.

새 API:

```http
PUT /api/sessions/current/questions
Content-Type: application/json
```

요청:

```json
{
  "questions": [
    { "text": "첫 번째 질문" },
    { "text": "두 번째 질문" }
  ]
}
```

성공 응답:

```json
{
  "questions": [
    { "priority": 1, "text": "첫 번째 질문", "draftThought": null },
    { "priority": 2, "text": "두 번째 질문", "draftThought": null }
  ]
}
```

검증:

- `questions` 길이는 trim 이후 비어 있지 않은 질문 기준으로 2~5개여야 한다.
- 각 `text`는 trim 후 비어 있으면 안 된다.
- 서버는 공백 앞뒤를 제거한 text를 저장한다.
- 요청에 5개 초과 질문이 있으면 `400 Bad Request`를 반환한다.
- 요청에 유효 질문이 2개 미만이면 `400 Bad Request`를 반환한다.

저장:

- 현재 인증 멤버의 현재 `OPEN` 세션을 찾는다.
- 트랜잭션 안에서 해당 멤버의 현재 세션 질문을 삭제한다.
- 요청 순서대로 priority `1..N`을 부여해 새로 insert한다.
- `draft_thought`는 새 UX에서 사용하지 않으므로 `null`로 저장한다.
- 현재 `OPEN` 세션이 없으면 기존 정책과 동일하게 current session not open 오류를 반환한다.

이 교체 저장 방식은 사용자가 질문 수를 줄이거나 순서를 정리했을 때 서버에 낡은 priority 질문이 남는 문제를 막는다.

## 10. Frontend API Design

새 action을 추가한다.

```ts
export async function saveQuestions(questions: Array<{ text: string }>) {
  return fetch("/api/bff/api/sessions/current/questions", {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ questions }),
  });
}
```

기존 `saveQuestion(priority, text, draftThought)`는 다른 호출 가능성을 위해 유지한다. 현재 세션 컴포넌트는 새 `saveQuestions`를 사용한다.

저장 전 프런트 검증:

- trim된 질문이 2개 미만이면 `질문은 최소 2개 작성해 주세요.`를 표시한다.
- trim된 질문이 5개 초과되는 UI 상태는 만들지 않는다.
- 저장 중에는 기존처럼 저장 버튼을 disabled 처리한다.
- 저장 실패 시 기존 `질문 저장 실패 · 다시 시도해 주세요` 피드백을 유지한다.

## 11. Error Handling

프런트 인라인 오류:

- 최소 질문 수 미달: `질문은 최소 2개 작성해 주세요.`
- 최대 질문 수 도달: 추가 버튼 주변에 `최대 5개까지 작성할 수 있어요.`

서버 오류:

- `400`: 프런트에는 저장 실패 상태를 보여준다. 최소 수 미달은 프런트에서 먼저 잡으므로 일반 사용자는 보기 어렵다.
- `401`: 기존 인증 흐름을 따른다.
- 현재 열린 세션 없음: 기존 current session 오류 정책을 따른다.
- 네트워크 실패: 기존 저장 실패 피드백을 유지한다.

부분 저장은 제공하지 않는다. 컬렉션 저장은 성공 또는 실패 단위로 처리한다.

## 12. Testing Plan

Frontend unit tests:

- 데스크탑에서 질문 입력칸이 기본 2개 렌더링되는지 확인한다.
- 모바일에서 질문 입력칸이 기본 2개 렌더링되는지 확인한다.
- 저장된 질문이 3개 있으면 3개 입력칸이 priority 순서대로 채워지는지 확인한다.
- `질문 추가`가 5개까지만 입력칸을 늘리는지 확인한다.
- `질문 3 삭제`가 세 번째 질문을 제거하고 번호를 다시 정리하는지 확인한다.
- 질문이 1개만 채워진 상태에서 저장하면 fetch가 호출되지 않고 최소 2개 오류가 보이는지 확인한다.
- 질문 2개 이상 저장 시 `PUT /api/bff/api/sessions/current/questions`로 `{ questions: [...] }`가 전송되는지 확인한다.
- `초안 생각 · 선택` 입력이 데스크탑과 모바일에 렌더링되지 않는지 확인한다.
- `My status`와 모바일 `내 상태`에서 `초안` 항목이 사라졌는지 확인한다.
- 데스크탑과 모바일 모두 한줄평이 서평보다 먼저 렌더링되는지 확인한다.
- 한줄평 저장 버튼이 primary 버튼 클래스를 사용하는지 확인한다.

Backend tests:

- `PUT /api/sessions/current/questions`가 질문 2개를 저장하고 priority 1, 2로 반환하는지 확인한다.
- 질문 5개 저장을 허용하는지 확인한다.
- 질문 1개 요청을 `400`으로 거절하는지 확인한다.
- 질문 6개 요청을 `400`으로 거절하는지 확인한다.
- 기존 질문 5개가 있는 상태에서 질문 2개를 PUT하면 DB에 2개만 남는지 확인한다.
- 저장된 `draft_thought`가 `null`인지 확인한다.
- 현재 열린 세션이 없으면 기존 current session not open 오류가 나는지 확인한다.

E2E smoke:

- 멤버가 현재 세션 화면에서 질문 2개를 작성해 저장한다.
- 새로고침 후 두 질문이 유지되고 공동 보드에도 반영된다.
- 모바일 뷰에서 질문 추가와 저장이 동작한다.

## 13. Migration and Compatibility

DB migration은 필요하지 않다. `questions.priority`는 이미 1~5를 허용하고, `draft_thought`는 nullable이다.

기존 `POST /api/sessions/current/questions`는 유지한다. 이 변경은 현재 세션 화면의 새 UX가 `PUT` 컬렉션 API를 사용하도록 추가하는 방식이다.

기존 저장 데이터에 `draftThought`가 있어도 현재 세션 질문 작성 UI에서는 보여주지 않는다. 이번 변경에서는 아카이브와 공동 보드의 기존 초안 표시를 유지한다.

## 14. Acceptance Criteria

- `/app/session/current` 데스크탑 질문 작성에서 `Q1~Q5` 선택 UI가 사라진다.
- `/app/session/current` 모바일 질문 작성에서 `Q1~Q5` 선택 UI가 사라진다.
- 질문 입력칸은 기본 2개이며 최대 5개까지 늘어난다.
- 질문은 최소 2개가 아니면 저장되지 않는다.
- 질문 저장은 현재 입력 목록 전체를 서버에 반영한다.
- 질문을 5개에서 2개로 줄이고 저장하면 서버에도 2개만 남는다.
- `초안 생각 · 선택` 입력은 데스크탑과 모바일에서 보이지 않는다.
- 상태 카드에서 `초안` 항목이 보이지 않는다.
- 한줄평 섹션은 서평 섹션보다 먼저 보인다.
- 한줄평 섹션은 서평 섹션과 같은 카드 톤을 사용한다.
- 관련 프런트와 백엔드 테스트가 통과한다.
