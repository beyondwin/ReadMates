## 목표

`v1.7.0 후속 하드닝`(2026-05-12 머지) 이후 진행한 심층 감사(서버·BFF·프론트·배포 파이프라인)에서 검증된 결함·논리·구조 항목을 한 사이클로 정리한다. 사용자 가시 동작은 보존하면서 (1) 보안/감사 무결성을 끌어올리고, (2) 클라이언트 권한 가드의 silent bypass를 닫고, (3) 시크릿 비교·권한 합성·릴리스 manifest 같이 두 군데에 흩어진 로직을 한 곳으로 모으고, (4) 후속 운영 자동화의 출발점이 되는 ADR/스키마를 정착시킨다.

## DEF-001: BFF audit executor graceful shutdown

### 현재 사실

- `server/src/main/kotlin/com/readmates/auth/infrastructure/security/BffSecretAuditExecutorConfig.kt:13`에서 `ThreadPoolTaskExecutor`를 만들 때 `setWaitForTasksToCompleteOnShutdown(false)`로 설정.
- `corePoolSize=1`, `maxPoolSize=2`, `queueCapacity=1000`.
- 종료 시 처리 중인 task와 큐에 남아 있는 task가 즉시 폐기됨. JVM이 graceful shutdown 신호를 받아도 audit row가 디스크에 닿지 않을 수 있음.
- 회전 진행 중 재기동(롤링 배포)이 가장 audit이 필요한 순간인데, 그 순간 손실 가능성이 가장 큼.

### 변경 후 동작

- shutdown 시 진행 중인 task가 완료될 때까지 대기. 최대 대기 시간은 명시적 타임아웃(기본 5초).
- 타임아웃 도달 시점에 큐에 남아 있던 task 수를 `bff.audit.shutdown.dropped` 카운터 메트릭으로 노출. 평상시 0이어야 함.
- 외부 동작(audit 기록 의미, header 흐름)은 변하지 않음.

### 구현 설계

- `BffSecretAuditExecutorConfig`를 다음과 같이 조정:
  - `setWaitForTasksToCompleteOnShutdown(true)`
  - `setAwaitTerminationSeconds(5)`
  - `setRejectedExecutionHandler { _, _ -> droppedCounter.increment() }` (queueCapacity 초과 시에도 카운트)
- `MeterRegistry` 의존성을 받아 `Counter.builder("bff.audit.shutdown.dropped").register(meterRegistry)` 등록.
- 메트릭 카탈로그(`docs/operations/observability/`)에 신규 카운터 한 줄 추가.

### 테스트 기준

- 새 단위 테스트 `BffSecretAuditExecutorConfigTest`:
  - executor가 queue capacity를 초과할 때 카운터가 증가한다.
  - `awaitTerminationSeconds`가 5인지 확인.
- 기존 `BffSecretFilterAuditTest`가 그대로 통과.

### 수용 기준

- `setWaitForTasksToCompleteOnShutdown(true)`이고 명시적 타임아웃이 설정되어 있다.
- 새 메트릭이 등록되어 `actuator/metrics`에서 조회 가능.
- 회귀 테스트가 모두 통과.

## DEF-002: ClientIpHashing baseSecret 필수화

### 현재 사실

- `server/src/main/kotlin/com/readmates/shared/security/ClientIpHashing.kt`의 `hashClientIp(raw, baseSecret, clock)`은 빈 `baseSecret`을 받아도 hash를 생성. 결과는 `"::2026-W18::ip"` 식의 약한 입력.
- 호출처 `BffSecretFilter:30` 및 `RateLimitFilter:30`이 `READMATES_IP_HASH_BASE_SECRET` 기본값을 `""`로 둠. 빈 값이면 startup 경고만 남기고 진행.
- 결과는 SHA-256을 `.take(32)`(hex 32자=128 bit)로 자름. baseSecret이 비어 있으면 IPv4 공간 + 주간 소금만으로 역산 가능.

### 변경 후 동작

- 운영 프로파일(`spring.profiles.active` 미설정 또는 production 포함)에서 `READMATES_IP_HASH_BASE_SECRET`가 비어 있으면 startup 실패.
- 테스트 / dev 프로파일에서는 명시적 `readmates.security.ip-hash.allow-empty-secret=true`인 경우에만 허용 + 단일 시작 로그 한 줄.
- hash 결과 길이 정책(`take(32)` = 128 bit truncation)은 유지하되, baseSecret이 비어 있지 않다는 사실 자체가 입력의 unpredictability를 보장.

### 구현 설계

- `ClientIpHashing`을 객체에서 클래스(또는 인터페이스 + 구현)로 전환하지 않고, 호출 시점에 baseSecret 검증을 호출자(`BffSecretFilter`, `RateLimitFilter`)에서 강제.
  - 더 간단한 옵션: `ClientIpHashing.hashClientIp`에 `requireNonBlankSecret: Boolean = true` 파라미터 추가. 빈 값이면 `IllegalStateException`.
- 호출자(`BffSecretFilter`, `RateLimitFilter`)에서 `@Value`로 받은 `ipHashBaseSecret`을 비워 둘 수 있는 경로를 다음 한 곳(`ClientIpHashingProperties`)에 모음:
  - 신규 `@ConfigurationProperties(prefix = "readmates.security.ip-hash")` 클래스 `ClientIpHashingProperties(val baseSecret: String, val allowEmptySecret: Boolean = false)`.
  - `@PostConstruct` 또는 정적 `validate()`에서 production 프로파일 + `allowEmptySecret=false` + blank → `IllegalStateException`.
- `application.yml`/`application-test.yml`에 새 키 추가:
  - production용 키는 기존 환경 변수 `READMATES_IP_HASH_BASE_SECRET`을 그대로 받음.
  - 테스트 yml은 다음 중 하나로 명시: (a) `readmates.security.ip-hash.allow-empty-secret=true`로 blank secret 허용, 또는 (b) 비-공백 `base-secret: test-ip-hash-base-secret` + `allow-empty-secret: false`로 실제 secret을 제공. (b)는 더 운영-유사 경로를 테스트하므로 선호. 두 경우 모두 `allowEmptySecret=true + blank` 코드 경로는 unit test로 별도 검증되어야 함.

### 테스트 기준

- `ClientIpHashingPropertiesTest`:
  - production 프로파일 + blank baseSecret → `IllegalStateException`.
  - dev/test 프로파일 + `allow-empty-secret=true` + blank → 통과.
  - 비공백 secret이 양쪽 모두에서 통과.
- 기존 `ClientIpHashingTest`는 회귀 없음(빈 secret 시나리오를 제거하고 명시적 `requireNonBlankSecret=false`로 우회).

### 수용 기준

- 프로덕션 환경에서 환경 변수 누락 시 의미 있는 startup 에러 메시지(현재 환경 변수 이름, 어떤 yml 키로 우회하는지 안내).
- 모든 환경에서 hash 결과 포맷(길이 32 hex)이 보존됨.
- 호출처 두 곳(`BffSecretFilter`, `RateLimitFilter`)이 동일 properties bean을 사용하므로 정책 표류가 발생하지 않음.

## DEF-003: route-guards clubSlug bypass 제거

### 현재 사실

- `front/src/app/route-guards.tsx:39-41`: `RequireMemberApp`가 `if (clubSlug) return <>{children}</>;` 로 `canUseMemberApp(state.auth)` 검증을 건너뜀.
- `route-guards.tsx:58-60`: `RequireHost` 동일 패턴(`canUseHostApp` skip).
- 결과: SUSPENDED, INACTIVE, 비-멤버가 `/clubs/<slug>/app/...` 경로로 직진 진입 → 서버가 응답을 막아도 클라이언트 단에서 권한 피드백 없음. 모바일/저속 회선에서 빈 UI/로딩만 보임.

### 변경 후 동작

- `RequireMemberApp` / `RequireHost`가 clubSlug 유무와 관계없이 항상 권한 함수(`canUseMemberApp` / `canUseHostApp`)를 실행.
- clubSlug 존재 여부는 `useParams()` validation 책임으로만 분리 — 권한 결정과 분리.
- 거절 시:
  - `RequireMemberApp`: 기존과 동일하게 `<BlockedMemberApp />` 표시.
  - `RequireHost`: 회원 앱 root(`scopedAppPath(clubSlug)`)로 `<Navigate>` 리다이렉트.
- 비-membership 상태의 사용자가 클럽 컨텍스트 안에 들어왔을 때 즉시 적절한 안내를 보게 됨.

### 구현 설계

- `RequireMemberApp` 함수 본문에서 `if (clubSlug) return <>{children}</>;` 제거.
- `RequireHost`의 동일 줄 제거.
- `canUseMemberApp` / `canUseHostApp`이 SUSPENDED / INACTIVE / 비-멤버를 거절하는지 단위 테스트로 확인. 기존 정의는 `front/shared/auth/member-app-access.ts`에 있으므로 import 변경 없음.

### 테스트 기준

- 신규 단위 테스트 `route-guards.test.tsx` 시나리오(이미 일부 존재할 수 있으면 보강):
  - SUSPENDED 멤버 + clubSlug 있음 → `BlockedMemberApp` 렌더.
  - INACTIVE 멤버 + clubSlug 있음 → `BlockedMemberApp` 렌더.
  - ACTIVE 멤버 + clubSlug 있음 → children 렌더.
  - 비-인증 + clubSlug 있음 → `<Navigate to={loginPath}>` 발생.
  - `RequireHost` 동일 매트릭스(MEMBER, GUEST, ACTIVE HOST 등).

### 수용 기준

- `front/tests/unit/route-guards.test.tsx`에 위 4~6 시나리오가 모두 PASS.
- 기존 E2E가 회귀 없이 통과(특히 host dashboard, member current session).
- 클라이언트 빌드 / lint / type check 깨끗.

## DEF-004: auth-context 401 분리 + redirect guard

### 현재 사실

- `front/src/app/auth-state.ts:5`: `AuthState = { status: "loading" } | { status: "ready"; auth: AuthMeResponse }`. 세션 만료를 표현할 상태 없음.
- `front/src/app/auth-context.tsx:30-34`: `fetchAuthMe().catch(() => setState({ status: "ready", auth: anonymousAuth }))`. 401 / 5xx / 네트워크 오류를 구분 없이 silent fallback.
- `front/shared/api/client.ts:49-54`: 모든 401에서 `window.location.assign(loginPathForReturnTo(currentRelativeReturnTo()))`. 짧은 시간 안에 여러 401을 받으면 무한 리다이렉트 가능.

### 변경 후 동작

- `AuthState`에 `{ status: "session_expired"; lastAuth?: AuthMeResponse }` 추가.
- `fetchAuthMe`가 응답 상태를 구별:
  - 200 → `{ status: "ready", auth }`.
  - 401 → `{ status: "session_expired", lastAuth: <이전 ready 상태> }`.
  - 그 외(네트워크/5xx) → `{ status: "ready", auth: anonymousAuth }` (현재 동작 유지).
- `readmatesFetchResponse`가 401 받았을 때:
  - `sessionStorage`(또는 module-level 변수) `LAST_LOGIN_REDIRECT_AT`을 검사. 마지막 redirect 후 1500ms 안에 다시 401이면 redirect 생략하고 `ReadMatesSessionExpiredError`를 throw.
  - throw된 에러는 호출자(예: route container)가 `session_expired` 상태로 처리해 사용자에게 명시적 안내.

## LOGIC-002: JdbcMemberLifecycle 상태 머신 명시

### 현재 사실

  - `suspendActiveMember`: ACTIVE → SUSPENDED.

  - `markMemberLeftByHost`: ACTIVE/SUSPENDED → LEFT.

### 변경 후 동작

- `server/.../auth/domain/MemberLifecycleStatus.kt` 신규 enum:
  ```kotlin
  enum class MemberLifecycleStatus(val storageValue: String) {
      ACTIVE("ACTIVE"),
      SUSPENDED("SUSPENDED"),
      LEFT("LEFT"),
      INACTIVE("INACTIVE");

      fun allowsTransitionTo(next: MemberLifecycleStatus): Boolean = TRANSITIONS[this].orEmpty().contains(next)

      companion object {
          private val TRANSITIONS: Map<MemberLifecycleStatus, Set<MemberLifecycleStatus>> = mapOf(
              ACTIVE to setOf(SUSPENDED, LEFT, INACTIVE),
              SUSPENDED to setOf(ACTIVE, LEFT, INACTIVE),
              LEFT to emptySet(),
              INACTIVE to emptySet(),
          )
  ```

## STRUCT-001: SecretComparator 공통화

### 현재 사실

- `BffSecretFilter.findMatchingSecretIndex`(server/.../security/BffSecretFilter.kt:127)와 `RateLimitFilter.secretMatchesAny`(server/.../security/RateLimitFilter.kt:169)가 같은 의미를 따로 구현.
- BFF 필터는 index를 반환(alias 해석용), Rate-limit 필터는 boolean을 반환.
- 두 함수의 timing-uniform 조건이 코드 리뷰에서 따로 검증되어야 하므로 회귀 위험.

### 변경 후 동작

- `server/.../shared/security/SecretComparator.kt` 신규 object/class:
  ```kotlin
  object SecretComparator {
      fun firstMatchingIndex(provided: ByteArray, candidates: List<ByteArray>): Int {
          var matched = -1
          candidates.forEachIndexed { idx, candidate ->
              if (MessageDigest.isEqual(provided, candidate)) matched = idx
          }
          return matched
      }

      fun matches(provided: ByteArray, candidates: List<ByteArray>): Boolean =
          firstMatchingIndex(provided, candidates) >= 0

      fun firstMatchingIndex(provided: String, candidates: List<String>): Int =
          firstMatchingIndex(
              provided.toByteArray(StandardCharsets.UTF_8),
              candidates.map { it.toByteArray(StandardCharsets.UTF_8) },
          )

      fun matches(provided: String, candidates: List<String>): Boolean =
          firstMatchingIndex(provided, candidates) >= 0
  }
  ```
- `BffSecretFilter.findMatchingSecretIndex`를 제거하고 `aliasFor`에서 `SecretComparator.firstMatchingIndex(provided, secrets)` 직접 호출.
- `RateLimitFilter.secretMatchesAny`를 제거하고 `SecretComparator.matches(provided, trustedBffSecrets)` 직접 호출.

### 구현 설계

- 신규 파일 + 두 필터에서 호출 교체.
- 두 필터의 외부 동작은 보존.

### 테스트 기준

- 신규 `SecretComparatorTest`:
  - 비매칭 시 -1.
  - 매칭 시 해당 index.
  - 단일 매칭(다른 모든 후보 길이/내용 변경에도 결과 안정).
  - String overload 회귀.

### 수용 기준

- 신규 ADR-0014(`docs/development/adr/0014-bff-secret-rotation-lifecycle.md`)이 다음을 정착:
  - 회전 단계 정의(staging, candidate, primary, retire).
  - 안전한 순서: server에 새 secret 추가(이미 multi-secret 허용) → BFF 환경 변수에 새 secret을 secondary로 추가 → BFF 재배포 → 새 secret을 primary로 승격 → 옛 secret 제거.
  - 각 단계별 cutoff 시간 권장(예: secondary 머지 후 24h 후에만 primary 승격).
- BFF에 `BFF_SECRET_ROTATION_STAGE` 환경 변수(기본 `stable`) 도입:
  - `stable` → 첫 항목만 사용(현재 동작).
  - `staging` → 첫 항목 사용하되, secondary 항목 사용 비율 일정(예: 10% 샘플링)을 디버깅 헤더 `X-Readmates-Bff-Secret-Index`로 표시(서버 audit과 교차 검증).
  - 본 plan에서는 `staging` 분기는 코드 hook만 만들고, 실제 샘플링 구현은 별도 후속(고도화 영역).
- 신규 BFF 진단 endpoint `GET /api/bff/__internal/secret-status`가 다음을 반환(JSON):
  ```json
  {"configuredSecretCount": 2, "primarySecretFingerprint": "sha256-prefix-6", "rotationStage": "stable"}
  ```
  - fingerprint는 secret 자체가 아니라 SHA-256의 첫 6자(공개 안전).

### 구현 설계

- ADR 본문.
- `front/functions/_shared/proxy.ts`에 `getRotationStage(env)`, `secretFingerprint(secret)` helper 추가.
- 신규 `front/functions/api/bff/__internal/secret-status.ts`(또는 동등 라우트). 인증 없이 호출 가능하지만 fingerprint만 노출.
- 테스트 fixture 갱신.

### 테스트 기준

- 신규 `cloudflare-bff-secret-status.test.ts`:
  - 단일 secret → `configuredSecretCount=1`, `rotationStage=stable`.
  - 멀티 secret → `configuredSecretCount=N`.
  - 비공개 fingerprint 형식(6자 hex prefix) 검증.
- 기존 BFF 테스트(`cloudflare-bff.test.ts`) 회귀 없음.

### 수용 기준

- ADR-0014 정착.
- `__internal/secret-status` 라우트가 동작하고 secret 자체를 노출하지 않음.
- BFF 테스트 통과.

## STRUCT-004: public-release-candidate 빌더 deny-list 우선화

### 현재 사실

- `scripts/build-public-release-candidate.sh`이 manifest를 명시 등록 방식(approved manifest path)으로 받음.
- 새 top-level 디렉토리(예: `terraform/`, `e2e-helpers/`) 추가 시 매번 빌더 수정 없으면 candidate에서 누락. silent miss 가능.
- 이미 `is_forbidden_candidate_path`(deny list)는 존재. allow와 deny가 둘 다 있는 hybrid 모델.

### 변경 후 동작

- 빌더가 다음 모델로 정렬:
  1. 모든 top-level git-tracked 경로를 후보로 수집.
  2. denylist(`node_modules`, `.git`, `.tmp`, `.gradle`, 기타 비공개 디렉토리, 알려진 secret 파일 패턴) 적용해 제거.
  3. 남은 모든 경로는 candidate tree에 포함.
- allow-list가 필요한 특수 케이스(예: `docs/private/`)는 명시적 denylist 항목으로만 표현.
- 새 fixture: 모든 top-level이 candidate에 포함되는지 자동 검증.
- 위 변경은 운영자가 의도적으로 비공개로 두고 싶은 디렉토리를 빌더에 명시 등록하도록 강제(누락이 silent가 아닌 fixture 실패로 드러남).

### 구현 설계

- `build-public-release-candidate.sh`:
  - `git ls-files`로 모든 추적 경로를 시작 집합으로.
  - 함수 `is_forbidden_candidate_path` 유지 + 확장(현재의 forbidden patterns + 신규 명시 denylist 디렉토리 목록).
  - approved manifest 함수(`is_approved_manifest_path`)는 deny에 걸리지 않은 모든 경로를 자동 승인하도록 단순화.
  - 신규 fixture(`scripts/fixtures/public-release-candidate-coverage.txt`)에 현재 후보 top-level 디렉토리 목록을 기록. `verify-public-release-fixtures.sh`가 빌더 출력과 fixture를 diff.

### 테스트 기준

- 새 top-level dir(예: `tools/sample-stub/`) 추가 후 빌더 실행 → candidate tree에 포함됨 + fixture diff 신호.
- 알려진 비공개 디렉토리(`.tmp/`, `node_modules/`)는 여전히 제외.
- public-release-check(gitleaks) 회귀 없음.

### 수용 기준

- 새 top-level dir 추가 시 빌더 코드 수정 없이 candidate에 자동 포함(명시 deny에 걸리지 않는 한).
- fixture 검증이 새 top-level 출현을 잡아냄.
