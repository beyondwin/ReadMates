# ReadMates Language, UX, And Data Consistency Design

작성일: 2026-04-22

## 1. 목적

ReadMates의 모바일/데스크탑 웹 화면에는 같은 기능을 다르게 부르는 문구, 영어/한글이 섞인 라벨, 실제 데이터처럼 보이지만 하드코딩된 UI, 입력은 있지만 저장되지 않는 값이 일부 남아 있다.

이 설계의 목적은 다음 문제를 한 번의 구현 플랜으로 정리하는 것이다.

1. 사용자-facing 용어를 한국어 중심으로 통일한다.
2. 모바일과 데스크탑에서 같은 기능이 같은 정보 구조와 이름으로 보이게 한다.
3. API 또는 DB 데이터로 연결되어야 하는 값이 정적 상수나 샘플처럼 보이지 않게 한다.
4. 호스트가 멤버 화면과 호스트 화면을 명확하게 전환할 수 있게 한다.
5. 빈 상태, disabled 상태, 접근성 라벨을 실제 동작과 맞춘다.
6. 멤버십 접근 언어를 게스트, 둘러보기 멤버, 정식 멤버, 호스트 단계로 맞춘다.

## 2. 범위

이번 작업은 UX 개선 수준 B로 진행한다. 전면 리디자인이나 새 설정 도메인 모델 구축은 하지 않는다.

포함:

- 화면 문구, 탭, 버튼, 도움말, 빈 상태, disabled 상태, 접근성 라벨 정리
- 공개 클럽, 멤버 홈, 현재 세션, 아카이브, 마이페이지, 호스트 대시보드, 호스트 세션 편집의 용어 통일
- 하드코딩된 운영 데이터와 API 연결 누락 감사 및 수정
- 호스트/멤버 작업 공간 전환 경로 보장
- 둘러보기 멤버와 정식 멤버의 읽기/쓰기 권한 문구 정리
- 관련 unit/e2e 테스트 갱신

제외:

- 완전한 i18n 프레임워크 도입
- 알림 설정, 테마 설정, 캘린더 연동, 탈퇴 기능의 신규 API 구현
- 새로운 디자인 시스템 또는 비주얼 리디자인
- DB에서 클럽 운영 소개 전체를 관리하는 CMS성 모델 구축
- 기존 public/member/host URL 구조의 대규모 변경

## 3. 현재 확인된 문제

### 3.1 언어 혼재와 네이밍 불일치

프론트 스캔에서 다음 유형이 확인되었다.

- 데스크탑 아카이브 상세의 `Summary`, `Club records`, `My records`, `Feedback`이 모바일의 `요약`, `기록`, `내 기록`, `피드백`과 다르다.
- `Member home`, `Operation timeline`, `Quick actions`, `Notifications`, `Preferences`, `Danger zone`, `Public sessions`, `Host note`, `Principles`, `The rhythm` 같은 영어 eyebrow가 한국어 화면 안에 섞여 있다.
- 같은 영역이 `마이`, `My`, `내 서가 · 계정`, `내 공간`처럼 다르게 불린다.
- 같은 세션 개념이 `세션`, `이번 세션`, `현재 세션`, `모임`, `회차`로 섞여 쓰인다.
- 접근성 라벨에 `App tabs`, `Notes filters`, `Selected session record counts` 같은 영어가 남아 있다.

### 3.2 데이터/API 연결 누락

다음 항목은 실제 데이터 또는 API 연결이 필요하다.

- `front/features/host/components/host-session-editor.tsx`
  - `시작 시간` 입력은 있지만 `buildHostSessionRequest`에 전달되지 않는다.
  - 기존 세션 편집 시 `session.startTime`으로 초기화하지 않고 기본 `20:00`을 사용한다.
  - 공개 설정의 `summary`와 `visibility` 상태가 `/api/host/sessions/{sessionId}/publication`에 연결되지 않는다.
  - 사이드바의 `마지막 저장 2분 전`, `저장됨`은 실제 저장 상태가 아니다.
- `front/features/public/components/public-club.tsx`
  - `/api/public/club`은 `clubName`, `tagline`, `about`, `stats`, `recentSessions`를 반환하지만 화면은 `CLUB_ABOUT`, `CLUB_HOST`, `CLUB_STARTED_AT` 등 정적 상수를 섞어 쓴다.
- `front/features/archive/components/my-page.tsx`
  - 알림, 개인 설정, 프로필 수정, 캘린더 연동, 테마, 탈퇴는 클릭 가능한 설정처럼 보이지만 API가 없다.
  - `fallbackRecentAttendances`가 실제 최근 참석 데이터가 없을 때 추정 막대를 만든다.
- `front/features/host/components/host-dashboard.tsx`
  - 운영 체크리스트와 일부 빠른 액션은 실제 진행 상태가 아니라 정적 배열이다.
  - 비활성 액션이 왜 비활성인지 충분히 설명하지 않는다.

### 3.3 호스트/멤버 전환 흐름

호스트는 멤버 권한에 운영 권한이 추가된 사용자다. 따라서 호스트는 멤버 화면도 계속 사용할 수 있어야 하고, 호스트 화면에서도 멤버 화면으로 돌아갈 수 있어야 한다.

현재 호스트 대시보드에는 데스크탑에서 `멤버 뷰로` 링크가 있으나, 전환 규칙이 전체 모바일/데스크탑 내비게이션에 일관되게 정의되어 있지 않다.

### 3.4 멤버십 접근 언어

초대 없이 Google로 로그인한 사용자는 승인 대기자가 아니라 둘러보기 멤버다. 둘러보기 멤버는 전체 비공개 세션 기록과 이번 세션 현황을 읽을 수 있지만, RSVP, 체크인, 질문, 서평 작성과 피드백 문서 열람은 제한된다.

초대 링크를 수락했거나 호스트가 전환한 계정은 정식 멤버다. 정식 멤버는 현재 세션 참여와 작성 기능을 사용할 수 있고, 참석한 회차의 피드백 문서를 읽을 수 있다. 공개 화면과 로그인 화면은 누구나 가입할 수 있다는 인상을 주지 않고, 정식 멤버 참여는 초대 링크 또는 호스트 전환으로 열린다고 설명한다.

## 4. 선택한 접근

선택한 접근은 "UX + 데이터 연결 감사 병행"이다.

문구 정리만 하면 화면은 깔끔해지지만, 입력값 누락과 API 미연결 문제는 남는다. 반대로 모든 설정을 DB/API 모델로 확장하면 범위가 커져 구현 플랜 하나로 다루기 어렵다.

따라서 이번 작업은 사용자에게 실제 동작처럼 보이는 화면부터 정리한다.

- 입력했는데 저장되지 않는 값은 저장되게 한다.
- 이미 존재하는 API가 있으면 UI를 연결한다.
- API가 없는 설정성 UI는 읽기 전용 또는 준비 중으로 명확히 표시한다.
- 제품 설명 문구는 정적으로 남길 수 있지만, 운영 데이터처럼 보이는 값은 샘플 fallback을 쓰지 않는다.

## 5. 용어 표준

한국어 중심으로 통일하고, 제품/표준 약어만 예외로 둔다.

허용 예외:

- Google
- Gmail
- RSVP
- PDF
- URL
- Passcode
- No.

표준 용어:

| 개념 | 표준 용어 | 사용 위치 |
|---|---|---|
| 앱 안의 기능 단위 | 세션 | 세션 편집, 세션 열기, 세션 정보 |
| 실제 참석 경험 | 모임 | 다음 모임, 모임 날짜, 모임 이후 |
| 번호가 붙은 기록/문서 | 회차 | 회차 피드백 문서, 회차별 기록 |
| 현재 준비 중인 세션 | 이번 세션 | 내비게이션, 주요 진입점 |
| 지금 열린 세션 상태 설명 | 현재 세션 | 화면 제목, empty state |
| 공개/개인 콘텐츠 상위 개념 | 기록 | 공개 기록, 내 기록 |
| 다른 멤버의 피드 | 클럽 노트 | 질문, 한줄평, 읽기 흔적 피드 |
| 지난 세션 저장소 | 아카이브 | 세션 목록, 내 기록, 피드백 문서 |
| 계정/내 활동 | 내 공간 | 멤버 내비게이션과 마이페이지 |
| 현재 피드백 기능 | 피드백 문서 | 멤버/호스트/아카이브 화면 |
| 운영 권한 영역 | 운영 | 호스트 대시보드 진입점 |
| 호스트 작업 공간 | 호스트 화면 | 전환 안내 문구 |
| 멤버 작업 공간 | 멤버 화면 | 전환 안내 문구 |
| 초대 없이 로그인한 읽기 전용 멤버 | 둘러보기 멤버 | 로그인, 멤버 홈, 호스트 멤버 관리 |
| 참여와 작성이 가능한 멤버 | 정식 멤버 | 로그인, 초대, 호스트 멤버 관리, 권한 안내 |

`report`는 URL 또는 코드 내부 호환 이름에는 남을 수 있지만 화면 문구에는 쓰지 않는다.

## 6. UX 구조

### 6.1 내비게이션

공개 영역:

- 소개
- 클럽
- 공개 기록
- 로그인

멤버 영역:

- 홈
- 이번 세션
- 클럽 노트
- 아카이브
- 내 공간

호스트 영역:

- 운영
- 세션 편집
- 멤버 초대
- 멤버 전환

모바일과 데스크탑은 화면 밀도와 배치는 달라도 같은 영역명과 같은 순서를 사용한다.

### 6.2 호스트/멤버 작업 공간 전환

호스트는 같은 로그인 세션에서 멤버 화면과 호스트 화면을 전환한다. 전환은 권한 변경이 아니라 라우트 이동이다.

규칙:

- 호스트가 `/app`에 있으면 멤버와 같은 홈/이번 세션/클럽 노트/아카이브/내 공간을 본다.
- 호스트에게만 멤버 화면에서 `운영` 또는 `호스트 화면` 진입점이 보인다.
- 호스트가 `/app/host` 계열에 있으면 운영 대시보드/세션 편집/초대/멤버 전환을 본다.
- 호스트 화면에는 항상 `멤버 화면으로` 돌아가는 액션이 있다.
- 모바일에서도 헤더 또는 하단 탭 주변에서 전환 경로를 잃지 않아야 한다.
- MEMBER 사용자는 호스트 전환 액션을 보지 못하고 호스트 라우트에 접근할 수 없다.

구현 대상:

- `front/shared/ui/top-nav.tsx`
- `front/shared/ui/mobile-header.tsx`
- `front/shared/ui/mobile-tab-bar.tsx`
- `front/features/member-home/components/member-home.tsx`
- `front/features/host/components/host-dashboard.tsx`

### 6.3 빈 상태와 disabled 상태

빈 상태 규칙:

- API 데이터가 비어 있으면 샘플을 보여주지 않는다.
- 다음 행동이 있으면 CTA를 함께 보여준다.
- 권한 때문에 제한된 경우는 이유와 다음 행동을 함께 보여준다.
- 비어 있는 목록에는 해당 화면의 표준 용어를 쓴다.

Disabled 상태 규칙:

- disabled 버튼은 이유를 설명한다.
- 아직 구현되지 않은 기능은 실제 설정처럼 보이지 않게 한다.
- API가 없는 설정은 "준비 중" 또는 읽기 전용 정보로 표시한다.

### 6.4 접근성 라벨

접근성 라벨은 보이는 문구와 같은 의미를 가져야 한다.

- 영어 `aria-label`은 한국어로 바꾼다.
- 아이콘 버튼은 기능을 구체적으로 설명한다.
- 탭/세그먼트 컨트롤은 같은 화면에서 모바일/데스크탑 라벨이 다르지 않게 한다.
- visible text와 aria label이 서로 다른 행동을 암시하지 않게 한다.

### 6.5 공개/로그인 접근 문구

공개 화면은 열린 가입처럼 보이는 문구를 쓰지 않는다.

- 공개 홈과 공개 세션 상세는 "읽는사이는 초대 기반 모임입니다. 공개 기록은 누구나 읽을 수 있고, 정식 멤버 참여는 초대 링크 또는 호스트 전환으로 열립니다."를 기준 문구로 쓴다.
- 로그인 화면은 기존 멤버 로그인 맥락을 먼저 보여주고, 초대 없이 로그인하면 둘러보기 멤버로 시작한다고 안내한다.
- 초대 수락 화면은 Google로 초대 수락하면 바로 정식 멤버가 된다고 안내한다.
- `PENDING_APPROVAL` 또는 승인 대기 표현은 화면 문구에서 쓰지 않는다. 호환 API 이름에만 남을 수 있다.

## 7. 데이터와 API 설계

### 7.1 호스트 세션 시간 저장

`시작 시간`은 실제 세션 데이터로 저장되어야 한다.

정책:

- 새 세션 기본값은 기존처럼 `20:00`을 사용할 수 있다.
- 기존 세션 편집은 `session.startTime`으로 초기화한다.
- 저장 payload에는 `startTime`을 포함한다.
- `endTime`은 현재 UI에 입력이 없으므로 기존 서버 기본 `22:00` 또는 기존 값을 유지한다.
- 질문 제출 마감 계산은 세션 날짜 기준 정책을 유지한다.

검증:

- 새 세션 저장 시 `startTime`이 payload에 들어간다.
- 기존 세션 편집 시 API의 `startTime`이 input 초기값으로 보인다.
- 시간을 바꾸고 저장하면 current session과 host detail에서 반영된다.

### 7.2 공개 설정 API 연결

호스트 편집의 공개 설정은 이미 존재하는 `PUT /api/host/sessions/{sessionId}/publication`에 연결한다.

정책:

- 세션 기본 정보 저장과 공개 설정 저장은 분리한다.
- 기존 publication이 있으면 `summary`와 `visibility` 초기값을 API 데이터에서 채운다.
- 공개 설정 UI는 현재 서버 API가 표현할 수 있는 두 가지 행동을 중심으로 정리한다.
- `요약 초안 저장`은 publication row를 저장하되 `isPublic=false`로 둔다. 공개 페이지에는 노출되지 않는다.
- `공개 기록 발행`은 `isPublic=true`로 저장한다. 서버 정책에 따라 닫힌 세션은 `PUBLISHED` 상태가 된다.
- `내부 공개`은 세션 기본 정보의 기본 상태 설명으로만 남기고, publication 저장 액션처럼 보이지 않게 한다.
- 공개 요약이 비어 있으면 publication API의 validation 메시지를 화면에 표시한다.
- 공개 저장 성공/실패는 일반 세션 저장 toast와 구분한다.

주의:

- 서버의 `PublicationRequest`는 현재 `publicSummary`와 `isPublic`만 받는다. UI도 이 두 값으로 표현 가능한 저장/발행 행동만 제공한다.
- 기존 UI의 `요약만 공개`은 public route 노출처럼 오해될 수 있으므로 이번 작업에서는 `요약 초안 저장`으로 문구를 바꾼다.
- 이미 공개된 기록을 다시 완전한 내부 상태로 되돌리는 기능은 별도 unpublish/delete-publication 정책이 필요하므로 이번 범위에 넣지 않는다.

### 7.3 공개 클럽 화면 데이터 정리

`/api/public/club`에서 이미 받는 값은 API 값을 우선 사용한다.

정책:

- `clubName`, `tagline`, `about`, `stats`, `recentSessions`는 API 값을 source of truth로 쓴다.
- `CLUB_ABOUT`은 제거하거나 API `about` fallback으로만 사용한다.
- `CLUB_HOST`, `CLUB_HOST_SINCE`, `CLUB_STARTED_AT`, `CLUB_CADENCE`, `CLUB_RECORDING`, `HOST_NOTE`는 아직 API에 없으므로 "정적 운영 소개"로 분리한다.
- 정적 운영 소개는 실제 DB 데이터처럼 보이는 표기보다 클럽 운영 안내 문구로 표현한다.
- 공개 화면의 영어 eyebrow는 한국어로 정리한다.

후속 과제:

- 클럽 운영 소개를 DB에서 관리하려면 별도 `club_profile` 또는 `club_public_settings` 설계가 필요하다. 이번 범위에서는 만들지 않는다.

### 7.4 마이페이지 설정성 UI 정리

마이페이지의 알림/개인 설정/탈퇴는 API가 없으므로 실제 설정처럼 보이지 않게 한다.

정책:

- 알림 토글은 클릭 가능한 switch처럼 보이지 않게 읽기 전용 상태로 바꾸거나 "준비 중" 배지를 붙인다.
- `변경`, `프로필 수정`, `탈퇴` 버튼은 동작이 없으면 disabled 처리하고 이유를 표시한다.
- 모바일의 `캘린더 연동`, `테마 · 표시`는 준비 중 항목으로 명확히 표시한다.
- `fallbackRecentAttendances`로 추정 참석 막대를 만들지 않는다. API의 `recentAttendances`가 비면 빈 상태를 보여준다.

### 7.5 호스트 대시보드 정적 운영 UI 정리

호스트 대시보드는 실제 운영 판단에 쓰이는 화면이므로 정적 상태를 줄인다.

정책:

- `Needs attention` 지표는 API 값을 유지하되 라벨을 한국어로 통일한다.
- 운영 체크리스트의 완료 여부는 계산 가능한 범위에서 current session과 dashboard metric으로 파생한다.
- 계산할 수 없는 항목은 완료/대기로 단정하지 않고 안내 항목으로 표시한다.
- 빠른 액션 중 실제 이동 가능한 항목은 링크로 유지한다.
- 아직 API가 없는 리마인더 발송은 disabled 상태와 이유를 표시한다.
- 저장 상태 카드의 `2분 전`, `저장됨`은 실제 상태가 아니므로 제거하거나 현재 세션 메타 정보로 대체한다.

## 8. 컴포넌트와 코드 경계

새로운 큰 추상화를 만들기보다 작은 helper를 추가한다.

예상 단위:

- `front/shared/ui/readmates-copy.ts` 또는 유사 helper
  - 공통 용어, nav label, empty-state label을 모은다.
- `front/features/host/components/host-publication-settings.tsx`
  - 공개 설정 API 연결과 상태 피드백을 host editor에서 분리한다.
- `front/features/host/components/host-dashboard-status.ts`
  - 운영 체크리스트/상태 파생 로직을 pure helper로 분리한다.
- `front/features/public/components/public-club-copy.ts`
  - 정적 운영 소개와 API fallback 규칙을 분리한다.

이 분리는 선택 사항이다. 구현 중 기존 파일에 작은 변경으로 충분하면 별도 파일을 만들지 않는다. 다만 공개 설정 API 연결처럼 상태와 네트워크가 늘어나는 부분은 host editor 파일이 다시 커지지 않도록 분리한다.

## 9. 오류 처리

- publication API가 실패하면 공개 설정 영역 안에서만 오류를 보여준다.
- 세션 저장 실패와 공개 설정 저장 실패는 서로 다른 메시지를 사용한다.
- start time 저장 실패는 기존 세션 저장 실패 흐름을 따른다.
- public club API 값이 비어 있으면 안전 fallback을 쓰되 샘플 운영 데이터로 채우지 않는다.
- 설정 API가 없는 UI는 사용자가 클릭해도 조용히 실패하지 않게 한다. 가능하면 클릭 불가능하게 만들고 이유를 보여준다.
- 둘러보기 멤버가 쓰기 또는 피드백 문서에 접근하면 정식 멤버에게 열리는 기능이라는 안내를 보여준다.

## 10. 테스트 계획

프론트 unit/component tests:

- 공개 클럽 화면이 API `about/tagline/stats`를 우선 렌더링한다.
- 로그인, 초대, 공개 화면이 둘러보기 멤버와 정식 멤버 접근 문구를 렌더링한다.
- 공개 클럽 화면에 기존 하드코딩 소개가 API 값을 덮어쓰지 않는다.
- 호스트 세션 편집이 `session.startTime`으로 초기화된다.
- 호스트 세션 저장 payload에 `startTime`이 포함된다.
- 공개 설정 저장이 publication API를 호출한다.
- publication 저장 실패/validation 실패가 공개 설정 영역에 표시된다.
- 마이페이지 설정성 UI가 동작하지 않는 버튼처럼 보이지 않는다.
- recentAttendances가 없을 때 추정 막대를 렌더링하지 않는다.
- 호스트 대시보드 checklist/quick action 라벨이 한국어 표준을 따른다.
- 데스크탑과 모바일 아카이브 상세 탭 라벨이 같은 의미를 쓴다.

라우팅/권한 tests:

- HOST 사용자가 `/app`에서 멤버 홈을 볼 수 있다.
- HOST 사용자가 멤버 화면에서 `/app/host`로 이동할 수 있다.
- HOST 사용자가 호스트 화면에서 `/app`로 돌아갈 수 있다.
- MEMBER 사용자는 호스트 전환 액션을 보지 못한다.
- MEMBER 사용자는 호스트 라우트 접근이 차단된다.

E2E smoke:

- 호스트 dev login 후 멤버 화면에서 운영 화면으로 이동하고 다시 멤버 화면으로 돌아온다.
- 호스트가 세션 시작 시간을 바꾸고 저장하면 현재 세션 화면에 반영된다.
- 호스트가 공개 요약을 저장하면 host detail 재진입 시 값이 유지된다.

검증 명령:

```bash
pnpm --dir front exec tsc --noEmit
pnpm --dir front lint
pnpm --dir front test
pnpm --dir front test:e2e
pnpm --dir front build
./server/gradlew -p server test
```

E2E는 로컬 MySQL과 브라우저 의존성이 준비된 경우 실행한다. 실행할 수 없으면 환경 blocker를 기록한다.

## 11. 구현 순서

1. 용어 표준과 접근성 라벨을 작은 범위부터 정리한다.
2. 호스트/멤버 작업 공간 전환 경로를 데스크탑과 모바일에서 보장한다.
3. 호스트 세션 시작 시간 저장 누락을 수정한다.
4. 공개 설정을 publication API에 연결한다.
5. 공개 클럽 화면의 API 값 우선 렌더링과 정적 운영 소개 분리를 적용한다.
6. 마이페이지의 동작하지 않는 설정성 UI와 recent attendance fallback을 정리한다.
7. 호스트 대시보드의 정적 checklist/quick action/save-state 표현을 정리한다.
8. 모바일/데스크탑 정보 구조와 라벨 일관성을 최종 점검한다.
9. focused tests를 먼저 돌리고, 마지막에 전체 checks를 실행한다.

## 12. 위험과 대응

- 문구 변경은 테스트 assertion을 많이 건드릴 수 있다. 의미 단위 테스트로 갱신하고 불필요한 exact text coupling은 줄인다.
- 멤버십 접근 문구가 섞이면 사용자가 공개 기록 열람과 정식 멤버 참여를 혼동할 수 있다. 공개 화면은 초대 기반 모임을 명시하고, 앱 안에서는 둘러보기 멤버와 정식 멤버를 일관되게 구분한다.
- 기존 공개 설정의 `요약만 공개` 표현은 public route 노출과 혼동될 수 있다. 이번 구현에서는 "요약 초안 저장"으로 문구를 조정해 오해를 줄인다.
- 설정성 UI를 disabled/read-only로 바꾸면 사용자가 기능 축소로 느낄 수 있다. 대신 실제 동작하지 않는 버튼을 남기는 것보다 신뢰성이 높다.
- 호스트 내비게이션을 바꾸면 모바일 탭의 공간이 부족할 수 있다. 작업 공간 전환은 탭을 늘리기보다 헤더/홈 액션으로 제공하는 쪽을 우선한다.
- 기존 dirty worktree가 있을 수 있으므로 구현 시 unrelated changes는 건드리지 않는다.

## 13. 완료 기준

- 사용자-facing 화면에서 허용된 예외를 제외한 영어 라벨 혼재가 줄어든다.
- 같은 기능은 모바일과 데스크탑에서 같은 용어로 보인다.
- 둘러보기 멤버와 정식 멤버 접근 단계가 로그인, 초대, 공개 화면, 문서에서 같은 의미로 설명된다.
- `시작 시간` 입력값이 저장되고 다시 불러와진다.
- 공개 설정이 publication API와 연결된다.
- API 데이터가 있는 공개 클럽 값은 정적 상수보다 우선한다.
- API가 없는 설정성 UI는 실제로 동작하는 것처럼 보이지 않는다.
- 호스트는 멤버 화면과 호스트 화면을 양방향으로 전환할 수 있다.
- MEMBER는 호스트 전환 액션을 보지 못한다.
- 관련 unit/e2e와 전체 frontend/backend checks가 통과하거나, e2e blocker가 명확히 기록된다.
