## RISK-P-002: BffSecretFilter 죽은 코드(`matchesAny`) 제거

### 현재 사실

- `BffSecretFilter.kt:126-137`의 `matchesAny(provided, candidates)`는 정의되어 있으나 클래스 어디에서도 호출되지 않는다.
- 실 매칭은 `aliasFor(provided)`에서 직접 `MessageDigest.isEqual` 호출로 이루어진다(`BffSecretFilter.kt:139-151`).
- 따라서 `matchesAny`는 어떤 의도(timing-uniform 비교 helper)를 가졌지만 통합되지 않은 상태로 남아 있다.

### 변경 후 동작

- 시크릿 매칭 로직은 단일 helper(`findMatchingSecretIndex`, RISK-P-003에서 도입)에 모이고, `matchesAny` 정의는 제거된다.
- 외부 동작 변화는 없다.

### 구현 설계

- `BffSecretFilter.matchesAny` 메서드를 통째로 삭제한다.
- 코드 검색으로 잔존 참조가 없음을 확인한다(`rg -n "matchesAny" server/src/main/kotlin/com/readmates/auth`).

### 테스트 기준

- 기존 `BffSecretFilterTest`, `BffSecretFilterUnitTest`, `BffSecretFilterAuditTest`가 모두 그대로 통과해야 한다.

### 수용 기준

- `matchesAny`를 참조하는 코드가 0건이다.
- 컴파일과 테스트가 통과한다.

## RISK-P-003: BffSecretFilter.aliasFor 타이밍 일관성

### 현재 사실

- `aliasFor(provided)`는 `secrets.forEachIndexed { idx, candidate -> if (MessageDigest.isEqual(...)) return ... }` 패턴으로 첫 매치에서 즉시 반환한다.
- 비-유효 시크릿에서는 모든 후보를 끝까지 비교한 뒤 `null`을 반환한다.
- 따라서 *유효한 시크릿을 가진 클라이언트* 사이에서 alias index에 따라 응답 시간이 달라진다. 외부 공격자가 자기 시크릿이 어떤 alias index에 매핑되는지 추론할 수 있다.
- 실제 보안 영향은 낮다(유효 시크릿 보유자만 자기 alias index를 알게 되며, 회전 기간 등 정보 가치도 제한적). 그러나 `RateLimitFilter.secretMatchesAny`는 이미 전수 비교 패턴을 사용한다. 정책을 통일하는 것이 코드 일관성과 회귀 방지 측면에서 적절하다.

### 변경 후 동작

- 시크릿 매칭은 항상 모든 candidate를 끝까지 `MessageDigest.isEqual`로 비교한다.
- 매칭된 candidate의 index만 누적하고, 최종 alias 매핑은 루프 종료 후 한 번 수행한다.
- 외부 동작(반환 alias 문자열, 거절 응답)은 변경되지 않는다.

### 구현 설계

`BffSecretFilter` 내부에 private helper를 도입한다.

```kotlin
private fun findMatchingSecretIndex(provided: String): Int {
    val providedBytes = provided.toByteArray(StandardCharsets.UTF_8)
    var matchedIndex = -1
    secrets.forEachIndexed { idx, candidate ->
        val candidateBytes = candidate.toByteArray(StandardCharsets.UTF_8)
        if (MessageDigest.isEqual(providedBytes, candidateBytes)) {
            matchedIndex = idx
        }
    }
    return matchedIndex
}

internal fun aliasFor(provided: String): String? =
    when (val idx = findMatchingSecretIndex(provided)) {
        -1 -> null
        0 -> "primary"
        1 -> "secondary"
        else -> "index_$idx"
    }
```

- `aliasFor`는 이제 매칭 루프를 직접 돌지 않고 helper의 결과만 매핑한다.
- `MessageDigest.isEqual`은 두 배열 길이가 다른 경우에도 constant-time(JDK 17+ 구현)이므로 추가 마스킹은 필요 없다.

### 테스트 기준

`server/src/test/kotlin/com/readmates/auth/infrastructure/security/BffSecretFilterUnitTest.kt`(또는 기존 unit test 파일)에 다음 시나리오를 추가한다.

- 첫 번째 candidate 매치 → `aliasFor`가 `"primary"` 반환.
- 두 번째 candidate 매치 → `"secondary"` 반환.
- 세 번째 이상 candidate 매치 → `"index_2"` 등 반환.
- 매치 없음 → `null` 반환.

추가로, 구현이 모든 candidate를 비교하는지 확인하는 보조 테스트는 다음 패턴을 사용한다.

- 호출 횟수를 계측 가능한 spy를 만들지 않고, 동일한 candidate를 N회 채운 설정으로 매치/비매치 시 동일한 cost 경로를 따르는지 코드 검토로만 검증한다(테스트 코드의 타이밍 측정은 회귀 신호로 신뢰할 수 없으므로 도입하지 않는다).

### 수용 기준

- `aliasFor`의 매칭 루프 내에서 `return` 문이 사라진다.
- 기존 외부 알리아스 문자열 의미("primary", "secondary", "index_N", null)가 보존된다.
- 신규/기존 BFF 시크릿 관련 테스트가 모두 통과한다.

## RISK-P-004: deploy-server.yml 이미지 빌드 일관성

### 현재 사실

- `.github/workflows/deploy-server.yml`은 두 번의 `docker/build-push-action` 호출을 가진다.
  - 첫 호출(line 70-77): `push: false`, `load: true`로 로컬 daemon에 적재.
  - 둘째 호출(line 96-105): `push: true`로 GHCR에 푸시.
- 두 호출 사이에 컨텍스트는 같지만, base image pull 시점이 다를 수 있어 Trivy가 스캔한 이미지의 digest와 GHCR에 push되는 이미지의 digest가 미세하게 달라질 가능성이 있다.

### 변경 후 동작

- 같은 build 호출 1회로 결과 이미지를 생성하고, 그 결과를 Trivy로 스캔한 후 GHCR에 push한다.
- Trivy 스캔이 실패하면 GHCR push가 실행되지 않는다.
- 이미지 digest 일관성이 GitHub Actions log에 명시적으로 보인다.

### 구현 설계

`docker/build-push-action`의 `outputs`에서 `digest`를 사용하는 패턴으로 단일화한다. 두 가지 후보 패턴 중 하나를 택한다.

**Option A: 단일 build → load → 스캔 → push (간단)**

```yaml
- name: Build server image for scan and push
  id: build
  uses: docker/build-push-action@bcafcacb16a39f128d818304e6c9c0c18556b85f # v7.1.0
  with:
    context: server
    file: server/Dockerfile.release
    platforms: linux/arm64
    push: false
    load: true
    tags: ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }}

- name: Scan image vulnerabilities
  uses: aquasecurity/trivy-action@ed142fd0673e97e23eac54620cfb913e5ce36c25 # v0.36.0
  with:
    image-ref: ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }}
    scanners: vuln
    format: table
    severity: HIGH,CRITICAL
    ignore-unfixed: true
    exit-code: '1'

- name: Push scanned image to GHCR
  uses: docker/build-push-action@bcafcacb16a39f128d818304e6c9c0c18556b85f # v7.1.0
  with:
    context: server
    file: server/Dockerfile.release
    platforms: linux/arm64
    push: true
    tags: ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }}
    sbom: true
    provenance: true
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

이 형태에서도 두 호출이지만 `cache-from`/`cache-to`로 두 번째 build가 첫 번째 결과를 그대로 재사용하도록 강제한다. 모든 레이어가 캐시 hit이면 최종 image digest는 첫 build와 동일하다.

**Option B: 단일 build로 tarball 생성 후 scan + skopeo push (확실)**

```yaml
- name: Build server image to tarball
  uses: docker/build-push-action@bcafcacb16a39f128d818304e6c9c0c18556b85f # v7.1.0
  with:
    context: server
    file: server/Dockerfile.release
    platforms: linux/arm64
    outputs: type=docker,dest=/tmp/readmates-server.tar
    tags: ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }}

- name: Load image into local daemon
  run: docker load --input /tmp/readmates-server.tar

- name: Scan loaded image
  uses: aquasecurity/trivy-action@ed142fd0673e97e23eac54620cfb913e5ce36c25 # v0.36.0
  with:
    input: /tmp/readmates-server.tar
    scanners: vuln
    format: table
    severity: HIGH,CRITICAL
    ignore-unfixed: true
    exit-code: '1'

- name: Push to GHCR
  run: |
    docker tag ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }} \
               ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }}
    echo "${{ secrets.GITHUB_TOKEN }}" \
      | docker login ghcr.io -u "${{ github.actor }}" --password-stdin
    docker push ${{ steps.image.outputs.name }}:${{ steps.image.outputs.tag }}
```

Option B는 SBOM/provenance attestation을 별도 단계로 분리해야 하므로 채택하지 않는다.

**채택**: Option A. `cache-from`/`cache-to: gha`로 두 번째 build가 첫 build의 모든 레이어를 재사용하도록 강제한다. 첫 build 결과로 Trivy가 스캔한 digest는 push되는 이미지의 digest와 일치한다(레이어가 모두 캐시 hit이면 동일 digest). 캐시 miss로 두 번째 build가 다른 digest를 만들면 attestation 단계에서 digest를 log로 노출해 회귀를 감지한다.

추가로 push 후 `outputs.digest`를 GitHub job summary에 기록한다.

```yaml
- name: Record pushed digest
  run: echo "Pushed digest=${{ steps.push.outputs.digest }}" >> "$GITHUB_STEP_SUMMARY"
```

### 테스트 기준

- workflow는 실제 GHCR push가 일어나야 검증 가능하므로 CI에서만 확인한다.
- PR 단계에서는 다음을 본다.
  - `actionlint`(존재하면) 또는 `gh workflow view`로 syntax 확인.
  - PR 본문에 다음 manual repro 노트를 남긴다.
    - 임시 tag(`v0.0.0-followup-test`)를 push해 workflow 동작과 digest 일관성을 확인.

### 수용 기준

- workflow에 build 단계가 2회 등장하더라도 양쪽 모두 동일한 컨텍스트/Dockerfile/플랫폼/태그를 사용한다.

## RISK-P-006: watch-compose-post-deploy.sh 에러 grep 정확도

### 현재 사실

- `deploy/oci/watch-compose-post-deploy.sh:53`에서 다음 패턴으로 최근 10분 로그를 검사한다.

```bash
if sudo docker compose -f compose.yml logs --since 10m readmates-api 2>/dev/null \
  | grep -E 'ERROR|Exception|Caused by' \
  | tail -120; then
  exit 1
fi
```

- 이 패턴은 다음 false positive를 유발할 수 있다.
  - 라이브러리/프레임워크가 정상 흐름의 일부로 출력하는 `ERROR-level이 아닌` 메시지 본문에 단어 `ERROR`가 포함될 때.
  - 정상 처리된 예외를 알리는 INFO/DEBUG 로그가 본문에 `Exception`을 포함할 때.
  - 스택트레이스 본문의 `Caused by:`는 항상 직전 ERROR-level 라인에 종속된다(따로 트리거할 필요가 없음).

### 변경 후 동작

- grep 패턴은 *log level이 ERROR인 라인*만 매칭한다.
- 스택트레이스(`Exception`, `Caused by:`)는 별도 트리거가 아니라, 이미 검출된 ERROR 라인의 컨텍스트로만 다룬다.
- 매칭이 있으면 deploy를 실패시키는 기존 동작은 유지된다.

### 구현 설계

Spring Boot 기본 logging pattern은 다음과 같다.

```
2026-05-12 03:14:15.123  ERROR 1 --- [main] c.r.SomeClass : message
```

- 로그 레벨 토큰은 좌우 공백으로 둘러싸인 단어이다.
- 따라서 다음 패턴을 사용한다.

```bash
grep -E '[[:space:]]ERROR[[:space:]]'
```

- 추가로 컨텍스트를 운영자에게 보이기 위해 `--max-count`를 사용하지 않고, 매칭된 라인을 그대로 tail로 자른다.

최종 블록은 다음과 같다.

```bash
echo "==> [watch] recent backend errors"
ssh "${SSH_OPTIONS[@]}" "${REMOTE_USER}@${VM_PUBLIC_IP}" \
  "sudo bash -s -- $(shell_quote "$REMOTE_DIR")" <<'EOF'
set -uo pipefail
remote_dir="$1"
cd "$remote_dir"
matches="$(sudo docker compose -f compose.yml logs --since 10m readmates-api 2>/dev/null \
  | grep -E '[[:space:]]ERROR[[:space:]]' \
  | tail -120 || true)"
if [ -n "$matches" ]; then
  printf '%s\n' "$matches"
  exit 1
fi
EOF
```

`set -e` 대신 `set -uo pipefail`을 사용하고, grep의 no-match exit 1이 변수 캡처를 중단하지 않도록 `|| true`를 둔다.

### 테스트 기준

- 로컬에서 다음 fixture로 패턴 동작을 확인한다.

```bash
printf '2026-05-12 03:14:15.123  ERROR 1 --- [main] c.r.A : boom\n' | grep -E '[[:space:]]ERROR[[:space:]]'
printf '2026-05-12 03:14:15.123  INFO  1 --- [main] c.r.A : retried ERROR-handler init\n' | grep -E '[[:space:]]ERROR[[:space:]]' && echo MATCH || echo SKIP
```

- 첫 명령은 매칭, 두 번째는 비매칭이어야 한다(`SKIP` 출력).
- `bash -n deploy/oci/watch-compose-post-deploy.sh`와 `shellcheck deploy/oci/watch-compose-post-deploy.sh`가 통과한다.

### 수용 기준

- ERROR 레벨이 아닌 라인은 deploy 실패를 유발하지 않는다.
- 실제 ERROR-level 라인이 발견되면 기존과 동일하게 watch 단계가 `exit 1`로 종료한다.
- `shellcheck` 경고 없음.

## RISK-P-007: 05-deploy 이미지 verification 진단 메시지 분리

### 현재 사실

- `deploy/oci/05-deploy-compose-stack.sh:199-207`에서 다음으로 running image id를 확인한다.

```bash
RUNNING_IMAGE_ID="$(
  ssh ... "cd ${REMOTE_DIR} && container=\$(sudo docker compose -f compose.yml ps -q readmates-api) && sudo docker inspect \"\$container\" --format '{{.Image}}'"
)"
if [ "$RUNNING_IMAGE_ID" != "$EXPECTED_IMAGE_ID" ]; then
  remote_ledger_append "IMAGE_VERIFIED" "FAILED" "expectedImageId=${EXPECTED_IMAGE_ID} runningImageId=${RUNNING_IMAGE_ID}"
  echo "Running readmates-api image mismatch: expected ${EXPECTED_IMAGE_ID}, got ${RUNNING_IMAGE_ID}" >&2
  false
fi
```

- `set -e`와 `$(...)`의 상호작용으로, SSH 실패 또는 `docker compose ps -q`가 빈 결과를 낼 때 `RUNNING_IMAGE_ID=""`가 되고, 비교가 mismatch로 처리된다.
- 운영자가 진단 메시지를 보면 "이미지가 다르다"고 오해할 수 있으나 실제 원인은 SSH 연결/inspect 실패일 수 있다.

### 변경 후 동작

- 빈 `RUNNING_IMAGE_ID`는 별도의 진단 메시지와 ledger event(`reason=running-image-id-empty`)로 보고한다.
- 비교는 양쪽 값이 모두 비어 있지 않을 때만 mismatch로 처리한다.
- 종료 코드는 둘 다 실패(`1`)지만, ledger detail이 다르므로 사후 분석에서 구분된다.

### 구현 설계

해당 블록을 다음으로 교체한다.

```bash
RUNNING_IMAGE_ID="$(
  ssh "${SSH_OPTIONS[@]}" "${REMOTE_USER}@${VM_PUBLIC_IP}" \
    "cd $(shell_quote "$REMOTE_DIR") && container=\$(sudo docker compose -f compose.yml ps -q readmates-api) && sudo docker inspect \"\$container\" --format '{{.Image}}'" \
    || true
)"
if [ -z "$RUNNING_IMAGE_ID" ]; then
  remote_ledger_append "IMAGE_VERIFIED" "FAILED" "reason=running-image-id-empty expectedImageId=${EXPECTED_IMAGE_ID}"
  echo "Failed to resolve running readmates-api image id via docker inspect" >&2
  false
fi
if [ "$RUNNING_IMAGE_ID" != "$EXPECTED_IMAGE_ID" ]; then
  remote_ledger_append "IMAGE_VERIFIED" "FAILED" "expectedImageId=${EXPECTED_IMAGE_ID} runningImageId=${RUNNING_IMAGE_ID}"
  echo "Running readmates-api image mismatch: expected ${EXPECTED_IMAGE_ID}, got ${RUNNING_IMAGE_ID}" >&2
  false
fi
remote_ledger_append "IMAGE_VERIFIED" "RUNNING" "imageId=${RUNNING_IMAGE_ID}"
```

- `|| true` 추가로 SSH 자체 실패에서도 진단 분기까지 도달한다(기존엔 `set -e` 우회로 동작했으나, 이 부분을 명시화).
- ledger의 `reason=` 필드는 기존 detail 포맷을 확장한 것이며 grep으로 식별 가능하다.

### 테스트 기준

- 다음 시나리오를 manual repro로 확인한다(실 인프라가 아닌 staging/임시 VM).
  - 정상 deploy 시 ledger에 `IMAGE_VERIFIED RUNNING imageId=...` 한 줄.
  - `readmates-api` 컨테이너가 떠 있지 않을 때 ledger에 `reason=running-image-id-empty` 명시.
  - 의도적으로 다른 이미지를 띄운 후 ledger에 `expectedImageId=... runningImageId=...` 명시.
- 자동화 측면에서는 `bash -n`과 `shellcheck`가 통과한다.

### 수용 기준

- 빈 image id와 mismatch가 별도의 ledger event detail로 구분된다.
- SSH/inspect 실패에서도 deploy는 명시적 실패로 종료한다.
- 정상 verification은 기존 ledger event(`IMAGE_VERIFIED RUNNING`)와 동일하게 기록된다.
