# Deploy Secrets via GitHub Actions — Design Spec

**Status:** Draft (awaiting user review)
**Author:** kws (with Claude)
**Date:** 2026-05-17

## 1. 배경 / 동기

현재 ReadMates 운영 시크릿은 OCI VM의 `/etc/readmates/readmates.env` 에 `deploy/oci/02-configure.sh` heredoc tee 로 1회 작성된다. 시크릿/설정값을 추가하거나 회전할 때마다 다음이 필요하다:

1. VM에 SSH 접속
2. `readmates.env` 직접 편집 (또는 `02-configure.sh` 수정 후 재실행)
3. `docker compose up -d --force-recreate readmates-api` 수동 실행

이 흐름은 (a) SSH 권한 보유자에 의존하고, (b) 변경 이력이 git에 안 남고, (c) 로컬 개발이 운영과 다른 메커니즘으로 동작해 "내 PC에서는 잘 됨" 격차를 만든다.

## 2. 목표 (Goals) / 비목표 (Non-Goals)

### Goals
- 시크릿/설정 추가·회전 시 **SSH 0회** — GitHub UI + Actions workflow_dispatch 한 번으로 완료
- **로컬 환경이 운영과 동일한 docker compose 메커니즘**으로 동작 (env 파일 출처만 다름)
- **GitHub·CI/CD 로그에 시크릿 값 노출 0건** (마스킹 + echo 금지)
- VM 저장 시크릿 파일 권한 최소화 (chmod 600)
- 변경 이력 추적 가능 (workflow run = actor + commit + timestamp)

### Non-Goals
- HashiCorp Vault / SOPS / sealed secrets 등 외부 시크릿 매니저 도입 (단일 VM·1인 운영에 과함)
- 다중 환경 (staging/preview) — 현재 단일 prod만 운영
- 시크릿 자동 회전 (runbook 기반 수동 회전)
- 셀프-호스티드 runner (GitHub-hosted runner로 충분)
- 레거시 host JAR 모드 유지 (사용자 확인: 더 이상 사용 안 함)

## 3. 아키텍처

```
[GitHub Repository Secrets / Variables]   ← single source of truth
              │
              ▼
.github/workflows/sync-config.yml         (workflow_dispatch only)
   │
   ├─ assert required secrets present (no values logged)
   ├─ render readmates.env via umask 077 heredoc → ${RUNNER_TEMP}
   ├─ scp (StrictHostKeyChecking=yes, known_hosts pinned)
   ├─ ssh: backup .bak; install -m 600; rm tmp
   └─ (optional) ssh: docker compose up -d --force-recreate readmates-api
              │
              ▼
[OCI VM]
  /etc/readmates/readmates.env   (chmod 600, owned by deploy user)
  /opt/readmates/compose.yml     (managed via deploy-server.yml/05-script — unchanged)

[로컬 dev]
  ./.env (gitignored — 사람이 .env.example로부터 복사 후 편집)
  docker compose -f deploy/oci/compose.yml -f deploy/local/compose.override.yml up
```

## 4. 보안 모델

### 4.1 GitHub 측
- **Repository Secrets** 만 사용 (Variables는 비민감만). 모든 Secrets는 GitHub에서 암호화 저장 + 로그 자동 마스킹.
- 워크플로 트리거: `workflow_dispatch` 만 (push/PR 이벤트로는 트리거 안 됨).
- **Environment "production"** 보호:
  - Required reviewer = 본인 (수동 승인 게이트)
  - `deployment_branch_policy` = `main` 브랜치 + `v*` 태그만 허용
- 워크플로 내 시크릿 처리 규칙:
  - **`echo $SECRET`, `printenv SECRET`, `cat .env` 금지**
  - 존재 검증은 `[ -n "${VAR:-}" ]` 로만, 결과는 `present` / `missing` boolean 만 출력
  - heredoc 작성 시 `umask 077` 선행
  - 시크릿 파일 경로는 `${RUNNER_TEMP}` 내부 (job 종료 시 자동 정리)

### 4.2 SSH 측 (사용자 선택: 옵션 b)
- 비표준 포트 (`2222`) + 키-only 인증 (PasswordAuthentication no)
- **전용 ed25519 deploy key** — 사용자 개인 키와 분리, 1회 부트스트랩 후 GitHub Secret으로 등록
- `fail2ban` (이미 VM에 설치됨) — sshd jail 활성화 확인
- SSH 유저: `deploy` (전용 비-root 유저), 비밀번호 없음, sudo는 **2가지 명령에만** 한정:
  ```
  /usr/bin/install -m 600 -o deploy /tmp/readmates.env.* /etc/readmates/readmates.env
  /usr/bin/docker compose -f /opt/readmates/compose.yml *
  ```
- 클라이언트 측: `StrictHostKeyChecking=yes`, `UserKnownHostsFile=<from secret>`, `BatchMode=yes`.

### 4.3 VM 측
- `/etc/readmates/` 디렉토리: `mkdir -p`, `chmod 750`, `chown deploy:deploy`
- `/etc/readmates/readmates.env`: `chmod 600`, `chown deploy:deploy`
- 백업: 쓰기 직전 `cp readmates.env readmates.env.bak` (`chmod 600` 보존)
- 워크플로 실패 시: `mv readmates.env.bak readmates.env` 로 자동 롤백 + 재시작

### 4.4 코드/저장소 측
- `.gitignore` 검증 완료: `.env`, `.env.*`, `deploy/oci/*.env`, `*.pem`, `*.key` 커버
- spec/runbook 문서에 **시크릿 값 평문 절대 금지** — 예시는 `<placeholder>` 형태로만
- PR description / commit message 작성 가이드: 시크릿 키 이름은 OK, 값은 금지

## 5. 컴포넌트 (산출물)

### 5.1 `.github/workflows/sync-config.yml` (신규)

**입력:**
- `restart_api` (choice: `true` / `false`, default `false`) — env 파일만 동기화할지, 동기화 후 컨테이너 재시작할지

**Job:** `sync` on `ubuntu-latest`, environment `production`

**Steps:**
1. `Assert required secrets present` — 모든 Tier-1 시크릿 키 이름을 순회하며 `[ -n "${!VAR:-}" ]` 검증. 누락 시 누락된 키 이름만 출력 후 exit 1.
2. `Render env file` — `umask 077`; heredoc으로 `${RUNNER_TEMP}/readmates.env` 작성. 첫 줄은 audit 헤더 주석:
   ```
   # generated by sync-config.yml run=${GITHUB_RUN_ID} sha=${GITHUB_SHA} actor=${GITHUB_ACTOR} at=$(date -u +%FT%TZ)
   ```
3. `Setup SSH` — `webfactory/ssh-agent` action으로 `READMATES_DEPLOY_SSH_KEY` 로드; `READMATES_VM_KNOWN_HOSTS` 를 `~/.ssh/known_hosts` 에 기록.
4. `Backup current env on VM` — `ssh deploy@$HOST -p $PORT 'sudo cp /etc/readmates/readmates.env /etc/readmates/readmates.env.bak 2>/dev/null || true'`
5. `Upload new env` — `scp -P $PORT ${RUNNER_TEMP}/readmates.env deploy@$HOST:~/readmates.env.new`
6. `Install` — `ssh ... 'sudo install -m 600 -o deploy /home/deploy/readmates.env.new /etc/readmates/readmates.env && rm /home/deploy/readmates.env.new'`
7. `Restart (optional)` — `restart_api=true` 일 때:
   ```
   ssh ... 'sudo docker compose -f /opt/readmates/compose.yml up -d --force-recreate readmates-api'
   ```
   실패 시 다음 step에서 `.bak` 복원:
   ```
   sudo install -m 600 -o deploy /etc/readmates/readmates.env.bak /etc/readmates/readmates.env
   sudo docker compose -f /opt/readmates/compose.yml up -d --force-recreate readmates-api
   exit 1
   ```
8. `Cleanup runner` — `${RUNNER_TEMP}/readmates.env` 자동 삭제 (job 종료 시).

**`secrets` 매핑:** 모든 시크릿/변수는 step `env:` 블록으로만 주입 (run script 안에서는 `"$VAR"` 만 사용). 시크릿 값을 `with:` 입력으로 넘기지 않음.

### 5.2 `deploy/oci/02-configure.sh` (수정)

**제거:**
- `tee /etc/readmates/readmates.env` heredoc 블록 (현재 14개 env var 작성)
- `systemctl enable readmates-server.service` (레거시 host JAR 모드)
- `readmates-server.service` 파일 작성 블록

**유지:**
- `mkdir -p /etc/readmates`, `chown`, `chmod 750`
- Caddy 설치/설정
- 디렉토리 권한 셋업

**추가:**
- 첫 실행 시 `/etc/readmates/readmates.env` 파일이 없으면 "run sync-config workflow first" 안내 메시지 출력 후 exit 0 (인프라 셋업은 시크릿 없이 완료 가능)

**참고:** `deploy` 유저 생성 + sudoers 작성은 1회성이므로 `02-configure.sh` 가 아닌 §5.5 부트스트랩 runbook에서 처리.

### 5.3 `deploy/local/compose.override.yml` (신규)

**목적:** `deploy/oci/compose.yml` 을 base로 사용하면서, 로컬에서만 다른 부분 (포트 노출, 로컬 MySQL 이미지, 로그 볼륨) override.

**내용 (스케치):**
```yaml
services:
  readmates-api:
    ports:
      - "8080:8080"
    environment:
      SPRING_PROFILES_ACTIVE: dev
    volumes:
      - ./logs:/var/log/readmates
  mysql:
    image: mysql:8.4
    ports:
      - "${READMATES_LOCAL_MYSQL_PORT:-3306}:3306"
    environment:
      MYSQL_ROOT_PASSWORD: ${READMATES_LOCAL_MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ${READMATES_LOCAL_MYSQL_DATABASE}
      MYSQL_USER: ${READMATES_LOCAL_MYSQL_USERNAME}
      MYSQL_PASSWORD: ${READMATES_LOCAL_MYSQL_PASSWORD}
  redis:
    ports:
      - "6379:6379"
```

**사용법:** `docker compose -f deploy/oci/compose.yml -f deploy/local/compose.override.yml --env-file .env up`

### 5.4 `docs/operations/runbooks/secrets-management.md` (신규)

4단계 절차:
1. GitHub Settings → Secrets and variables → Actions → **New repository secret**, 키/값 입력
2. (신규 키일 때) `.env.example` 에 키 이름 + 주석 추가, `sync-config.yml` 의 env 매핑에 한 줄 추가, PR 머지
3. Actions 탭 → "sync-config" workflow → **Run workflow** (input: `restart_api=true`) → production env approval 클릭
4. `https://api.<domain>/actuator/health` 헬스체크 (`up`) 확인

**회전 절차도 동일** (Step 1에서 키 값만 교체).

### 5.5 `docs/operations/runbooks/vm-deploy-key-bootstrap.md` (신규)

1회 셋업 절차:
1. 로컬에서 ed25519 키 페어 생성: `ssh-keygen -t ed25519 -C "github-actions-deploy" -f ./readmates-deploy -N ""`
2. VM에서 `deploy` 유저 생성 (`adduser --disabled-password --gecos "" deploy`)
3. VM `/home/deploy/.ssh/authorized_keys` 에 pubkey 추가 (chmod 600, chown deploy:deploy)
4. VM `/etc/sudoers.d/readmates-deploy` 작성 (chmod 440):
   ```
   deploy ALL=(root) NOPASSWD: /usr/bin/install -m 600 -o deploy /home/deploy/readmates.env.new /etc/readmates/readmates.env
   deploy ALL=(root) NOPASSWD: /usr/bin/install -m 600 -o deploy /etc/readmates/readmates.env.bak /etc/readmates/readmates.env
   deploy ALL=(root) NOPASSWD: /usr/bin/cp /etc/readmates/readmates.env /etc/readmates/readmates.env.bak
   deploy ALL=(root) NOPASSWD: /usr/bin/docker compose -f /opt/readmates/compose.yml *
   ```
5. VM `/etc/ssh/sshd_config.d/99-readmates.conf` 에 비표준 포트 + 비밀번호 인증 비활성화:
   ```
   Port 2222
   PasswordAuthentication no
   PubkeyAuthentication yes
   PermitRootLogin no
   AllowUsers deploy <existing-admin-user>
   ```
   재시작: `systemctl restart ssh`. **주의:** 기존 admin 세션 유지한 채로 새 포트 접속 검증 후 기존 포트 닫기.
6. `fail2ban` sshd jail 활성화 확인 (`systemctl status fail2ban`, `fail2ban-client status sshd`).
7. VM에서 `ssh-keyscan -p 2222 -t ed25519 <host>` 출력을 로컬에 복사.
8. GitHub Repository Settings → Secrets:
   - `READMATES_DEPLOY_SSH_KEY` ← `cat readmates-deploy` (private key 내용)
   - `READMATES_VM_KNOWN_HOSTS` ← Step 7 출력
9. GitHub Repository Settings → Variables:
   - `READMATES_VM_HOST` = `<host>`, `READMATES_VM_USER` = `deploy`, `READMATES_VM_SSH_PORT` = `2222`, `READMATES_DEPLOY_ROOT` = `/opt/readmates`
10. 로컬에서 private key 파일 안전 삭제 (`shred -u readmates-deploy`).
11. sync-config workflow 1회 실행해 readmates.env 초기 배포.

### 5.6 `.env.example` (수정)

각 키에 주석으로 다음 두 가지 중 하나 표시:
- `# LOCAL ONLY — not synced to prod` (예: `READMATES_LOCAL_MYSQL_*`)
- `# PROD via GitHub Secret <KEY_NAME>` (모든 시크릿 값들)
- `# PROD via GitHub Variable <KEY_NAME>` (비민감 설정)

## 6. 데이터 흐름

### 6.1 시크릿 동기화 (workflow run)

```
GitHub Secrets ──env: 매핑──▶ workflow runner 환경변수
                                    │
                                    ▼
                       umask 077 + heredoc ──▶ ${RUNNER_TEMP}/readmates.env
                                    │
                                    ▼ scp (SSH agent + known_hosts pin)
                       deploy@vm:~/readmates.env.new
                                    │
                                    ▼ sudo install
                       /etc/readmates/readmates.env  (chmod 600, owner deploy)
                                    │
                                    ▼ (옵션) docker compose up -d --force-recreate
                       readmates-api 컨테이너 재시작 (env_file 리로드)
```

### 6.2 로컬 개발

```
.env.example ──사람이 복사 후 편집──▶ .env (gitignored)
                                          │
                                          ▼ docker compose --env-file .env
                       compose.yml + compose.override.yml ──▶ 로컬 컨테이너 스택
```

### 6.3 운영 컨테이너 부팅

```
docker compose up
  └─ env_file: /etc/readmates/readmates.env  ──▶ readmates-api 컨테이너 환경변수
                                                       │
                                                       ▼
                                          Spring Boot `${VAR:default}` 해석
```

## 7. 오류 / 실패 처리

| 시나리오 | 동작 |
|---|---|
| 필수 시크릿 누락 | step 1 (assert)에서 누락된 키 이름 출력 + exit 1. SSH 연결 없음. |
| SSH 연결 실패 | scp/ssh step에서 즉시 exit 1. VM 상태 변경 없음. |
| scp 중간 끊김 | 임시 파일은 `~/readmates.env.new` 에만 존재. atomic mv 전이므로 `/etc/readmates/readmates.env` 미변경. 다음 run에서 덮어씀. |
| docker compose up 실패 | `.bak` 자동 복원 + 재시작. workflow exit 1로 페이지. |
| `.bak` 복원도 실패 | workflow exit 1, GitHub 알림. 수동 SSH 개입 필요 (runbook: "비상 복구" 섹션). |
| 시크릿 값이 잘못된 형식 (예: API key 만료) | workflow는 성공 (env 동기화는 정상). 컨테이너 재시작 후 헬스체크에서 발견. 운영자는 헬스체크 + Sentry로 인지. |

## 8. 테스트 전략

### 8.1 정적 검증
- `actionlint` (이미 ci.yml에서 사용 중인지 확인 — 없으면 추가)
- `shellcheck` for `02-configure.sh`
- workflow 파일 로컬 dry-run: `act -j sync --secret-file ...` (선택, 문서화만)

### 8.2 통합 검증
- **스테이징 VM 없음** — 다음으로 신뢰 가능한 검증:
  1. workflow의 "render env file" step만 실행하는 `dry-run` 입력 추가 (input: `dry_run=true`). scp/ssh 안 함. 파일 내용은 로깅 안 함 (line count + sha256만 출력).
  2. dry-run 통과 후 실제 run.
- 부트스트랩 후 첫 실행은 readmates.env 사전 복사 본 떠놓고 진행.

### 8.3 로컬 검증
- `cp .env.example .env`, 값 채운 후 `docker compose -f deploy/oci/compose.yml -f deploy/local/compose.override.yml up` 으로 스택 기동.
- 헬스체크 `curl localhost:8080/actuator/health` = `UP`.

## 9. 마이그레이션 단계

1. **부트스트랩** (runbook 5.5 따라 deploy 유저 + 키 + sudoers + 비표준 포트 + secrets/variables 등록). 기존 SSH 접근은 유지.
2. **sync-config workflow 첫 실행** (dry-run → 실제). 기존 `/etc/readmates/readmates.env` 가 덮어써짐 → 사전에 키들이 모두 GitHub Secrets에 등록되어 있어야 함.
3. **검증**: 컨테이너 재시작 + 헬스체크 OK.
4. **02-configure.sh 정리** (시크릿/레거시 systemd 블록 제거). 향후 신규 VM 셋업 시 무 시크릿으로 동작 확인.
5. **runbook 게시** + 팀(=본인)에 알림.

## 10. 자주 묻는 질문

**Q: GitHub-hosted runner의 IP가 VM에 접근 가능해야 하는데, 방화벽은?**
A: OCI security list에서 포트 2222를 0.0.0.0/0 으로 개방 (키-only + fail2ban + 비표준 포트로 위험 완화). 향후 GitHub Actions IP ranges 만 허용하도록 좁힐 수 있음 (`api.github.com/meta` 의 `actions` 배열, 6시간 마다 변경 가능성 있음).

**Q: secret 이 GitHub Actions 로그에 마스킹되는데 어떻게 잘못 누출될 수 있나?**
A: 워크플로 안에서 시크릿을 base64로 인코딩하거나 다른 변수에 대입하면 마스킹 우회됨. **`env:` 매핑 외 변형 금지**.

**Q: PR description에 시크릿 키 이름 적어도 되나?**
A: 키 이름은 OK. 값은 금지. PR/issue 본문은 인덱싱되고 fork에 노출될 수 있음.

**Q: 로컬 .env 에 운영 시크릿 값을 복사해도 되나?**
A: 권장하지 않음. 로컬은 dev 값 (테스트 DB, 테스트 API 키)만. 운영 데이터에 접근해야 한다면 VM에서 직접 (감사 가능).

## 11. 향후 강화 항목 (Out-of-scope, 추후)

- Cloudflare Tunnel for SSH (제로트러스트, public 22/2222 노출 0)
- GitHub Actions IP allowlist를 SSH 방화벽에 자동 동기화하는 cron job
- 시크릿 만료/회전 알림 (예: API key 90일 회전 리마인더)
- Multi-environment (staging) — 별도 environment + 별도 VM
- 시크릿 사용 audit log 수집 (workflow run 외에 컨테이너 부팅 시 어떤 키를 읽었는지 기록)
