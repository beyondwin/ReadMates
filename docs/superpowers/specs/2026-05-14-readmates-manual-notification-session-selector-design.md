# ReadMates 수동 알림 세션 선택 컨트롤 설계

## 배경

호스트가 모바일 하단 탭 또는 데스크탑 운영 내비게이션에서 `알림 발송` 화면으로 바로 진입하면 수동 알림 워크벤치에 세션 컨텍스트가 없다. 현재 UI는 `세션 ID` 텍스트 입력칸을 노출하므로 운영자는 내부 UUID를 알아야 하고, 세션 편집 화면에서 진입하지 않으면 정상적인 발송 준비가 어렵다.

기존 서버와 프론트엔드에는 필요한 기반이 이미 있다.

- `GET /api/host/sessions`는 호스트 세션 목록을 반환한다.
- `GET /api/host/notifications/manual/options`는 선택된 `sessionId`가 있으면 세션 요약, 템플릿 활성화 상태, 멤버 대상 정보를 함께 반환한다.
- 수동 알림 미리보기와 발송은 이미 `sessionId` 기반 선택 모델을 사용한다.

따라서 새 기능은 API를 크게 확장하지 않고, 알림 발송 화면의 세션 선택 UX를 운영자가 사용할 수 있는 형태로 바꾸는 데 집중한다.

## 목표

- 하단 탭에서 `알림 발송`으로 바로 진입해도 세션 ID를 직접 입력하지 않고 수동 알림을 준비할 수 있게 한다.
- 모바일과 데스크탑 모두에서 선택된 세션을 명확히 확인할 수 있게 한다.
- URL에 `sessionId`가 포함된 기존 진입 경로는 유지한다.
- 세션 변경 시 템플릿 활성화 상태, 대상 멤버 목록, 세션 힌트가 선택된 세션에 맞게 갱신되게 한다.

## 비목표

- 세션 검색 API를 새로 만들지 않는다.
- 세션 선택 모달 또는 별도 세션 탐색 화면은 만들지 않는다.
- 수동 알림 발송 도메인 규칙, 중복 발송 규칙, 대상 산정 규칙은 변경하지 않는다.
- 알림 장부의 이벤트/배송/테스트 메일 탭 구조는 변경하지 않는다.

## 결정 사항

### 세션 목록 범위

알림 화면의 세션 선택 컨트롤은 호스트가 접근할 수 있는 전체 세션 목록을 보여준다. 정렬은 기존 `GET /api/host/sessions` 응답 순서를 따른다. 현재 서버 정렬은 운영 중인 세션을 먼저 보여주고, 그 뒤 초안 및 과거/완료 세션을 이어서 보여주는 구조다.

이 선택은 `모임 전날 리마인더`처럼 예정 세션이 필요한 경우와 `피드백 문서 등록`처럼 과거/완료 세션이 필요할 수 있는 경우를 모두 막지 않는다. 특정 템플릿이 선택된 세션에 부적합한지는 기존 수동 알림 옵션 응답의 `enabled`와 `disabledReason`으로 표현한다.

### 초기 선택

URL에 `sessionId`가 있고 그 값이 로더가 받은 호스트 세션 목록에 있으면 해당 세션을 우선 선택한다.

URL에 `sessionId`가 없거나 목록에 없는 값이면 목록의 첫 번째 `OPEN` 세션을 자동 선택한다. `OPEN` 세션이 없으면 목록의 첫 번째 세션을 선택한다. 세션 목록이 비어 있으면 선택 없음 상태를 보여주고 미리보기 버튼을 비활성화한다.

### UI 형태

기존 `세션 ID` 텍스트 입력칸을 제거하고, 같은 위치에 `세션 선택` 컨트롤을 둔다. 컨트롤은 세션 번호, 책 제목, 날짜를 함께 보여준다.

선택 컨트롤 아래 또는 옆에는 선택된 세션 요약을 보여준다.

- 데스크탑: `세션 선택` 컨트롤과 선택 요약을 워크벤치 내부에서 나란히 배치한다.
- 모바일: `세션 선택` 컨트롤 아래에 선택 요약을 쌓는다.

요약에는 세션 번호, 책 제목, 날짜, 상태, 공개 범위, 리마인더 기준 D-day 힌트를 포함한다. 피드백 문서 알림처럼 세션 조건이 맞지 않는 템플릿은 기존 템플릿 비활성 사유로 설명한다.

### 데이터 흐름

알림 라우트 로더는 기존 알림 장부 데이터와 함께 호스트 세션 목록을 불러온다.

1. 라우트 로더가 `GET /api/host/sessions`와 기존 알림 데이터를 병렬 조회한다.
2. URL `sessionId`가 있고 세션 목록에 같은 ID가 있으면 그 값을 초기 선택으로 사용한다.
3. URL `sessionId`가 없거나 세션 목록에 없으면 세션 목록에서 첫 번째 `OPEN` 세션을 고른다.
4. 선택된 세션 ID로 `manual/options`를 조회해 세션 요약, 템플릿 상태, 멤버 목록을 초기화한다.
5. 사용자가 세션을 변경하면 같은 `manual/options` 호출을 다시 실행한다.
6. 수동 알림 미리보기/발송 요청은 선택 컨트롤이 가진 `sessionId`를 사용한다.

선택 변경은 가능한 한 화면 안에서 처리한다. URL query 갱신은 필수 요구사항이 아니다. 단, URL에 이미 들어온 `sessionId`는 기존 호환성을 위해 계속 존중한다.

## 컴포넌트 영향

- `front/features/host/route/host-notifications-data.ts`
  - 호스트 세션 목록을 로더 데이터에 추가한다.
  - 초기 수동 알림 선택을 URL 또는 세션 목록 기반으로 결정한다.

- `front/features/host/route/host-notifications-route.tsx`
  - 로더에서 받은 세션 목록을 페이지 컴포넌트로 전달한다.

- `front/features/host/ui/host-notifications-page.tsx`
  - 수동 알림 워크벤치에 세션 목록과 선택 변경 콜백을 전달한다.
  - 세션 변경 시 기존 `onLoadManualOptions` 흐름으로 옵션 상태를 갱신한다.

- `front/features/host/ui/notifications/manual-notification-workbench.tsx`
  - `세션 ID` 텍스트 입력을 `세션 선택` 컨트롤로 교체한다.
  - 선택된 세션 요약을 모바일/데스크탑 반응형으로 보여준다.
  - 세션 변경 시 선택 상태, 재발송 확인, 미리보기 상태가 일관되게 초기화되도록 콜백을 호출한다.

## 오류 및 빈 상태

- 세션 목록 조회 실패는 라우트 로더 실패로 처리한다. 이는 기존 알림 화면의 초기 데이터 실패와 같은 성격이다.
- 선택한 세션의 `manual/options` 재조회가 실패하면 페이지 에러로 전환하지 않고 워크벤치 안의 인라인 오류를 보여준다.
- 세션이 없으면 `선택 가능한 세션이 없습니다` 상태를 보여주고 `미리보기`를 비활성화한다.
- URL에 존재하지 않는 `sessionId`가 들어오면 초기 로더가 페이지 전체를 실패시키는 대신 첫 가용 세션으로 복구한다. 세션 목록도 비어 있을 때만 선택 없음 상태를 보여준다. 이 동작은 구현 시 기존 `manual/options` 호출 순서를 조정해 라우트 에러를 피한다.

## 테스트 전략

- 단위 테스트
  - URL `sessionId`가 없을 때 첫 `OPEN` 세션이 초기 선택되는지 검증한다.
  - URL `sessionId`가 있으면 해당 세션이 우선 선택되는지 검증한다.
  - 세션 선택 변경 시 `onLoadManualOptions`가 새 세션 ID로 호출되고 미리보기 버튼 상태가 갱신되는지 검증한다.
  - 세션 목록이 비어 있으면 미리보기 버튼이 비활성화되는지 검증한다.

- E2E 테스트
  - 하단 탭 또는 직접 알림 화면 진입 후 세션 ID를 입력하지 않고 기본 `OPEN` 세션으로 미리보기를 만들 수 있는지 검증한다.
  - 세션 선택을 다른 세션으로 바꾼 뒤 미리보기가 선택된 세션 기준으로 생성되는지 검증한다.

- 회귀 확인
  - 기존 URL 기반 진입 경로가 유지되는지 검증한다.
  - 기존 수동 알림 발송 및 중복 발송 확인 흐름이 깨지지 않는지 검증한다.

## 승인된 방향

승인된 최종 방향은 `기존 수동 알림 워크벤치 안에 세션 선택 컨트롤과 선택 요약을 추가하는 방식`이다. 모바일에서는 세로로 쌓고, 데스크탑에서는 선택 컨트롤과 요약을 나란히 둔다. 하단 탭으로 들어온 경우 첫 번째 `OPEN` 세션을 자동 선택한다.
