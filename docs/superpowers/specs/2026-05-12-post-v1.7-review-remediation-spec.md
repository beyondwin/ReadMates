# v1.7.0 이후 전체 리뷰 Remediation 상세 스펙

작성일: 2026-05-12
상태: 구현 대기
연결 구현 문서: `docs/superpowers/plans/2026-05-12-post-v1.7-review-remediation-implementation-plan.md`

## 목표

`v1.7.0..HEAD` 전체 리뷰에서 확인된 release blocker, 테스트 회귀, auth/session UX 결함, 서버 lifecycle 계약 불일치, CI/deploy artifact 보증 약점을 한 번의 후속 remediation 사이클로 정리한다.

이 문서는 현재 코드와 검증 결과를 기준으로 "무엇이 결함인지", "어떤 동작이 정답인지", "어떤 검증이 완료 조건인지"를 고정한다. 실제 코드 변경 순서는 연결 구현 문서를 따른다.

## 검토 기준

- 기준 범위: `v1.7.0..HEAD`
- 현재 브랜치 상태: `main` ahead 상태의 누적 변경
- 적용 표면: frontend auth/BFF, server auth/security/lifecycle, GitHub Actions deploy, public-release scripts, CHANGELOG

## 상세 요구사항

### REL-001: Public release candidate secret scan 실패

#### 현재 사실

- `front/tests/unit/cloudflare-bff-secret-status.test.ts`의 legacy secret fixture가 scanner 관점에서 real-looking BFF secret assignment로 보이는 값을 사용한다.
- `.gitleaks.toml`의 `readmates-real-secret-assignment` 규칙은 `READMATES_BFF_SECRET`에 16자 이상 real-looking 값이 들어간 경우를 금지한다.
- 실제 실행 결과:
  - `./scripts/build-public-release-candidate.sh` 통과
  - `./scripts/public-release-check.sh .tmp/public-release-candidate` 실패
  - 실패 위치: `front/tests/unit/cloudflare-bff-secret-status.test.ts`

#### 변경 후 동작

- public release candidate에 포함되는 테스트 fixture는 scanner allowlist와 일치하는 public-safe test value만 사용한다.
- raw secret 비노출 테스트는 real-looking secret 문구 없이도 "응답에 입력값이 포함되지 않는다"를 검증한다.
- public release check와 gitleaks가 모두 통과한다.

#### 수용 기준

- `./scripts/build-public-release-candidate.sh` 통과.
- `./scripts/public-release-check.sh .tmp/public-release-candidate` 통과.
- `./scripts/verify-public-release-fixtures.sh` 통과.
- secret-status unit test는 기존 의미를 유지한다.

### TEST-001: Redis auth session cache test 고정 만료일

#### 현재 사실

- `RedisAuthSessionCacheAdapterTest.storedSession()`은 `expiresAt = 2026-05-12T00:00Z`를 고정한다.
- 현재 날짜가 2026-05-12라 `RedisAuthSessionCacheAdapter.find()`의 만료 검사에서 snapshot이 null이 된다.
- 실제 실행 결과:
  - `./server/gradlew -p server clean test` 실패
  - `stores and loads session snapshot`
  - `eviction failure records operation metrics without local cache bypass`

#### 변경 후 동작

- 테스트 fixture의 `expiresAt`은 테스트 실행일 기준 미래 시간이거나 충분히 먼 고정 미래일이어야 한다.
- 테스트는 cache codec/store/find 동작을 검증해야 하며, 달력 날짜 변화에 따라 실패하지 않아야 한다.

#### 수용 기준

- `./server/gradlew -p server test --tests com.readmates.auth.adapter.out.redis.RedisAuthSessionCacheAdapterTest` 통과.
- `./server/gradlew -p server clean test`에서 Redis auth session cache 관련 실패가 사라진다.

### SEC-001: IP hash blank secret 검증 gap

#### 현재 사실

- `ClientIpHashingProperties` KDoc은 non-production blank secret이 `allowEmptySecret=true`일 때만 허용된다고 설명한다.
- 실제 구현은 `dev`/`staging` 등 non-production profile에서 `baseSecret=""`, `allowEmptySecret=false`인 경우를 조용히 통과시킨다.
- production-like 판정은 active profile이 없거나 profile 문자열에 `production`이 포함된 경우다.

#### 변경 후 동작

- `baseSecret`이 blank이면 production-like profile에서는 `allowEmptySecret` 값과 무관하게 startup validation이 실패한다.
- non-production profile에서는 `allowEmptySecret=true`일 때만 blank secret을 허용하고 WARN을 한 번 남긴다.
- non-production profile에서 `allowEmptySecret=false`이면 명시적 예외로 실패한다.
- `BffSecretFilter`와 `RateLimitFilter`는 validator가 보장한 `baseSecret` 정책을 전제로 IP hash를 만든다.

#### 수용 기준

- production blank + `allowEmptySecret=false` 실패.
- production blank + `allowEmptySecret=true` 실패.
- dev/test blank + `allowEmptySecret=false` 실패.
- dev/test blank + `allowEmptySecret=true` 통과 + WARN.
- nonblank secret은 profile과 무관하게 통과.

### FE-001: `session_expired` route guard 무한 loading

#### 현재 사실

- `AuthProvider`는 `/api/bff/api/auth/me` 401을 `session_expired`로 저장한다.
- `RequireAuth`, `RequirePlatformAdmin`, `RequireMemberApp`, `RequireHost`는 `state.status !== "ready"`를 모두 loading으로 처리한다.
- `PendingApprovalRoute`도 `session_expired`를 loading으로 처리한다.
- loader가 없는 guard-only route에서는 세션 만료 후 사용자가 spinner에 머물 수 있다.

#### 변경 후 동작

- route guard는 `loading`과 `session_expired`를 분리한다.
- `session_expired`는 현재 route를 returnTo로 보존한 login route로 replace navigation한다.
- pending approval route도 `session_expired`를 loading으로 처리하지 않는다.
- 기존 `readmatesFetchResponse` redirect cool-off는 유지한다.

#### 수용 기준

- 각 guard에서 `session_expired` 상태가 loading label을 렌더하지 않는다.
- login route로 이동한다.
- route guard unit test에 `session_expired` matrix가 포함된다.
- E2E auth/member/host flow가 유지된다.

### API-001: Member lifecycle storage status 불일치

#### 현재 사실

- `MembershipStatus`는 `INVITED`, `VIEWER`, `ACTIVE`, `SUSPENDED`, `LEFT`, `INACTIVE`를 정의한다.
- `MemberLifecycleStatus`는 `ACTIVE`, `SUSPENDED`, `LEFT`, `INACTIVE`만 정의한다.
- `JdbcMemberLifecycleStoreAdapter.validateTransition()`은 DB에서 읽은 `memberships.status`를 `MemberLifecycleStatus.fromStorage()`로 변환한다.
- DB row가 `INVITED` 또는 `VIEWER`이면 `IllegalArgumentException`이 발생해 500으로 이어질 수 있다.

#### 변경 후 동작

- lifecycle transition validation은 storage에 존재할 수 있는 membership status 전체를 인식한다.
- `VIEWER`/`INVITED` row는 unknown status로 처리되지 않는다.
- host deactivate API는 service의 `MembershipStatus.canTransitionTo(LEFT)` 정책과 adapter SQL이 일치해야 한다.
- 불가능한 transition은 domain/application conflict로 매핑되고, unknown storage value만 진짜 invariant violation으로 남긴다.

#### 수용 기준

- `MemberLifecycleStatus.fromStorage("VIEWER")`와 `"INVITED"`가 성공한다.
- host가 viewer/invited member를 deactivate할 때 500이 나지 않는다.
- 선택된 정책에 따라 either:
  - row가 `LEFT`로 변경되고 200 응답, 또는
  - 명시적 lifecycle conflict로 409 응답.
- 본 스펙은 service의 기존 transition matrix와 맞춰 `VIEWER`/`INVITED -> LEFT` 성공을 선호한다.

### CI-001: Deploy server scanned artifact 보증

#### 현재 사실

- deploy-server workflow는 scan용 image를 한 번 build/load하고 Trivy scan을 실행한다.
- 그 뒤 release tag용 image를 다시 build/push한다.
- 두 build가 같은 Dockerfile/cache를 사용하더라도 scan한 digest와 push한 digest가 동일하다는 강제 보장이 없다.

#### 변경 후 동작

- workflow는 한 번 build한 manifest digest를 scan하고, 같은 digest를 release tag로 promote한다.
- final release tag는 scan을 통과한 digest를 가리킨다.
- step summary에는 scanned digest와 promoted release tag가 기록된다.

#### 수용 기준

- workflow YAML에 scan build와 release push의 별도 rebuild가 없다.
- Trivy scan target은 digest 기반 image ref다.
- release tag는 scan이 통과한 digest로 생성된다.

### DOC-001: CHANGELOG 코드 불일치

#### 현재 사실

- CHANGELOG는 BFF audit executor await termination을 10초로 적지만 코드는 5초다.
- CHANGELOG는 audit-mode 복원 값을 `full`로 적지만 코드는 `all`만 허용한다.
- CHANGELOG는 secret-status rotation stage를 `single`/`rotating`/`unconfigured`로 적지만 코드는 `stable`/`staging`만 반환한다.

#### 변경 후 동작

- CHANGELOG의 operator-facing 값은 코드, tests, ADR과 일치한다.
- 잘못된 env/config 값으로 운영자가 startup failure를 만들 가능성을 줄인다.

#### 수용 기준

- `awaitTerminationSeconds=5`
- `readmates.security.bff.audit-mode=all`
- `rotationStage="stable" | "staging"`
- `git diff --check -- CHANGELOG.md` 통과.

### BFF-001: Secret status endpoint cache policy

#### 현재 사실

- `/api/bff/__internal/secret-status`는 인증 없이 JSON을 반환한다.
- ADR은 인증 미필요를 수용한 비용으로 기록한다.
- 응답에 `Cache-Control`이 없어 edge/client cache가 최신 rotation state를 보여주지 못할 수 있다.

#### 변경 후 동작

- 응답 header에 `Cache-Control: no-store`를 포함한다.
- 기존 JSON body shape는 바꾸지 않는다.

#### 수용 기준

- unit test가 `cache-control` header를 검증한다.
- 기존 secret count/fingerprint/stage 테스트가 통과한다.
