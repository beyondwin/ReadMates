# ReadMates Toolchain And Dependency Refresh Design

## 1. Goal

ReadMates의 로컬 개발, CI, deploy, frontend workspace, server wrapper가 같은 명시적 도구 계약을 사용하도록 정렬하고, 2026-07-16 기준으로 확인된 최신 frontend 및 Gradle 버전으로 갱신한다.

이번 변경은 제품 기능이나 API를 바꾸지 않는다. 변경 후에도 frontend route/BFF/server architecture와 공개 저장소 안전 경계는 그대로 유지되어야 한다.

## 2. Current State

- CI와 deploy workflow는 Node.js 24를 사용하지만 저장소 루트에는 Node 버전 파일이 없다.
- 루트와 frontend `package.json`은 `pnpm@10.33.0`을 고정한다.
- frontend, design system, design docs는 하나의 pnpm workspace와 lockfile을 공유한다.
- frontend workspace에는 patch/minor 업데이트와 ESLint, TypeScript, jsdom major 업데이트가 함께 남아 있다.
- server Gradle wrapper는 9.1.0이고 Java toolchain은 JDK 25다.
- Corepack이 PATH에 없는 환경은 이미 `npx --yes corepack@0.35.0` fallback을 사용한다.

## 3. Target Contract

### 3.1 Runtime And Package Manager

- Node.js repository contract: major 24.
- Root and frontend package manager: `pnpm@11.13.1`.
- Corepack fallback: `corepack@0.35.0`; this version remains unchanged because it is the current registry version and already encoded in CI, deploy, scripts, and active docs.
- Add one repository Node version file and a root `engines` contract so version managers and package-manager diagnostics agree with CI.
- Do not change the user's machine-wide default Node installation. Validation must launch Node 24 explicitly when the current shell uses another major.

### 3.2 Frontend And Design Workspace

Upgrade the shared workspace coherently so packages that must stay aligned use one version:

- React and React DOM: 19.2.7.
- `@types/react`: 19.2.17.
- `@types/react-dom`: 19.2.3.
- Vite: 8.1.5.
- `@vitejs/plugin-react`: 6.0.3.
- Vitest and `@vitest/coverage-v8`: 4.1.10.
- Playwright test and component-test packages: 1.61.1.
- TanStack Query runtime and ESLint plugin: 5.101.2.
- React Router DOM: 7.18.1.
- `tsx`: 4.23.1.
- `globals`: 17.7.0.
- `eslint-plugin-react-refresh`: 0.5.3.
- `typescript-eslint`: 8.64.0 before evaluating the ESLint 10 compatibility lane.
- ESLint: 10.7.0.
- `@eslint/js`: 10.0.1.
- TypeScript: 7.0.2.
- jsdom: 29.1.1.

`@types/node` moves to 24.13.3 and remains on the Node 24 major because the repository runtime contract is Node 24. Major 26 types are out of scope.

All exact versions must be revalidated against the npm registry at implementation time. If a target has advanced, use the latest release within the target major and record the resolved version in the lockfile.

### 3.3 Server Build Tool

- Gradle wrapper: 9.6.1.
- Preserve JDK 25, Kotlin 2.4.0, Spring Boot 4.0.6, and existing server dependency constraints unless Gradle 9.6.1 requires a narrowly scoped compatibility repair.
- Regenerate wrapper properties and wrapper JAR through the wrapper task; do not hand-edit the JAR.
- Preserve the distribution SHA-256 contract.

## 4. Delivery Strategy

Use one isolated worktree and one feature branch, but keep four independently diagnosable stages:

1. Add the Node 24/pnpm 11 contract and update package-manager contract tests, docs, workflows, and lockfile.
2. Apply frontend/design patch and minor updates and repair only compatibility failures caused by those updates.
3. Apply ESLint 10, TypeScript 7, and jsdom 29 major updates separately. Retain each major only after its focused and workspace gates pass.
4. Upgrade the Gradle wrapper to 9.6.1 and run server wrapper/quality gates.

Each stage must leave the repository internally consistent. A failure in one major lane does not justify reverting already verified independent upgrades. An incompatible major may be deferred only with the exact failing command, error, attempted narrow repairs, and a pinned version that restores all gates.

## 5. Test And Verification Design

### 5.1 RED-First Contract Evidence

Before editing version contracts, add or update focused assertions that state the desired Node and pnpm values. Run them against the old tree and confirm they fail for the expected version mismatch. Configuration generated by package managers or Gradle is exempt from unit-level TDD, but every generated change must be verified by the owning tool.

Focused gates:

- package-manager and Docker CT command contract tests;
- agent-guidance self-test and active-doc version scans;
- frozen pnpm install under Node 24;
- frontend lint, unit coverage, build, Zod fixture freshness, and design workspace checks;
- targeted component-test Docker contract tests before the full component-test lane;
- Gradle wrapper version and checksum verification;
- `./scripts/server-ci-check.sh` and full Testcontainers integration tests.

### 5.2 Final Matrix

Run the following on the completed branch through the repository-defined launchers:

- Node 24 plus `pnpm@11.13.1` version proof;
- frozen workspace install;
- `pnpm --dir front lint`;
- `pnpm --dir front test:coverage`;
- `pnpm --dir front build`;
- `pnpm --dir front zod:export-fixtures` followed by a fixture diff check;
- `pnpm design:check`;
- `pnpm --dir front test:e2e`;
- `pnpm --dir front test:ct:docker`;
- `./scripts/server-ci-check.sh`;
- `./server/gradlew -p server integrationTest`;
- `./scripts/build-public-release-candidate.sh`;
- `./scripts/public-release-check.sh .tmp/public-release-candidate`;
- `git diff --check` and targeted public-safety/version scans;
- `graphify update .` with ignored output left unstaged.

If Docker, browser, or Testcontainers infrastructure is unavailable, report the exact skipped or failed command. Do not replace that evidence with a smaller check while claiming the full lane passed.

## 6. Documentation And Release Surface

Update active version claims in root/package guides, local setup, test guidance, scripts guidance, and package-manager contract fixtures. Historical `docs/superpowers/**` and past `CHANGELOG.md` verification records retain the versions that were true at their time and must not be rewritten.

Add one `Unreleased` CHANGELOG entry covering:

- Node 24 repository pin;
- pnpm 11 migration;
- frontend/design dependency refresh;
- Gradle wrapper refresh;
- validation evidence and any deferred major.

Generated `graphify-out/**`, build output, coverage, local caches, and runtime evidence remain untracked.

## 7. Failure Handling

- Lockfile or install failure: identify whether it is pnpm 11 format/policy drift or an incompatible dependency range before changing application code.
- Lint/type failure: make the smallest configuration or source compatibility repair and add focused regression evidence when behavior changes.
- Browser/component-test failure: distinguish package/browser binary mismatch from product regression and reinstall the exact Playwright browser version before editing tests.
- Gradle failure: verify the wrapper under JDK 25, then isolate plugin incompatibility by task. Do not upgrade unrelated server libraries unless required by a demonstrated Gradle 9.6.1 incompatibility.
- Public-release failure: treat new secrets, paths, generated artifacts, or scanner findings as release blockers.

## 8. Non-Goals

- Changing frontend behavior, UI, routes, API contracts, auth, persistence, or migrations.
- Moving CI from Node 24 to Node 26.
- Upgrading Java, Kotlin, Spring Boot, provider SDKs, or Gradle plugins without a direct compatibility requirement.
- Publishing, deploying, tagging, pushing, or merging the result.
- Rewriting historical plans, reports, or old CHANGELOG command evidence.

## 9. Acceptance Criteria

- Repository and CI package-manager contracts resolve Node 24 and pnpm 11.13.1 consistently.
- The shared lockfile installs frozen under Node 24.
- Retained frontend/design upgrades pass their complete relevant gates.
- Gradle 9.6.1 runs with JDK 25 and passes server PR-level and integration gates.
- Active docs and contract tests contain no stale pnpm 10.33.0 or Gradle 9.1.0 claim.
- Public-release and Graphify checks complete without tracked generated output.
- Any deferred major is documented with reproducible evidence and does not leave a partially migrated configuration.
